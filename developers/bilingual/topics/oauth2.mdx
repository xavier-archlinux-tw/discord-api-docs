## Shared Resources

The first step in implementing OAuth2 is [registering a developer application](https://discord.com/developers/applications) and retrieving your client ID and client secret. Most people who will be implementing OAuth2 will want to find and utilize a library in the language of their choice. For those implementing OAuth2 from scratch, please see [RFC 6749](https://tools.ietf.org/html/rfc6749) for details. After you create your application with Discord, make sure that you have your `client_id` and `client_secret` handy. The next step is to figure out which OAuth2 flow is right for your purposes.

## State and Security

Before we dive into the semantics of the different OAuth2 grants, we should stop and discuss security, specifically the use of the `state` parameter. [Cross-site request forgery](https://en.wikipedia.org/wiki/Cross-site_request_forgery), or CSRF, and [Clickjacking](https://en.wikipedia.org/wiki/Clickjacking) are security vulnerabilities that must be addressed by individuals implementing OAuth. This is typically accomplished using the `state` parameter. `state` is sent in the authorization request and returned back in the response and should be a value that binds the user's request to their authenticated state. For example, `state` could be a hash of the user's session cookie, or some other nonce that can be linked to the user's session.

When a user begins an authorization flow on the client, a `state` is generated that is unique to that user's request. This value is stored somewhere only accessible to the client and the user, i.e. protected by the [same-origin policy](https://en.wikipedia.org/wiki/Same-origin_policy). When the user is redirected, the `state` parameter is returned. The client validates the request by checking that the `state` returned matches the stored value. If they match, it is a valid authorization request. If they do not match, it's possible that someone intercepted the request or otherwise falsely authorized themselves to another user's resources, and the request should be denied.

While Discord does not require the use of the `state` parameter, we support it and highly recommend that you implement it for the security of your own applications and data.

## 狀態與安全性 (State and Security)

在我們深入研究不同 OAuth2 授權類型的語意之前，我們應該停下來討論安全性，特別是 `state` 參數的使用。跨站請求偽造 (CSRF) 和點擊劫持 (Clickjacking) 是實作 OAuth 的個人必須解決的安全性漏洞。這通常可以透過使用 `state` 參數來完成。`state` 在授權請求中發送，並在回應中傳回，它應該是一個將使用者的請求與其已驗證狀態綁定在一起的數值。例如，`state` 可以是使用者工作階段 (session) Cookie 的雜湊值，或者可以與使用者工作階段相關聯的其他一次性隨機數 (nonce)。

當使用者在用戶端開始授權流程時，會產生一個該使用者請求專屬的 `state`。此數值儲存在只有用戶端和使用者才能存取的地方，即受到[同源政策 (same-origin policy)](https://en.wikipedia.org/wiki/Same-origin_policy)保護。當使用者被重新導向時，會傳回 `state` 參數。用戶端藉由檢查傳回的 `state` 是否與儲存的數值相符來驗證該請求。如果相符，則是有效的授權請求。如果不相符，則可能有人攔截了請求，或者以其他方式將自己錯誤地授權給了另一位使用者的資源，該請求應被拒絕。

雖然 Discord 不強制要求使用 `state` 參數，但我們支援此功能，並強烈建議您為您自己應用程式和資料的安全性實作此參數。

## Authorization Code Grant

The authorization code grant is what most developers will recognize as "standard OAuth2" and involves retrieving an access code and exchanging it for a user's access token. It allows the authorization server to act as an intermediary between the client and the resource owner, so the resource owner's credentials are never shared directly with the client.

All calls to the OAuth2 endpoints require either HTTP Basic authentication or `client_id` and `client_secret` supplied in the form data body.

<ManualAnchor id="authorization-code-grant-authorization-url-example" />
###### Authorization URL Example

```
https://discord.com/oauth2/authorize?response_type=code&client_id=157730590492196864&scope=identify%20guilds.join&state=15773059ghq9183habn&redirect_uri=https%3A%2F%2Fnicememe.website&prompt=consent&integration_type=0
```

`client_id` is your application's `client_id`. `scope` is a list of [OAuth2 scopes](/developers/en-us/topics/oauth2#shared-resources-oauth2-scopes) separated by url encoded spaces (`%20`). `redirect_uri` is whatever URL you registered when creating your application, url-encoded. `state` is the unique string mentioned in [State and Security](/developers/en-us/topics/oauth2#state-and-security).

When someone navigates to this URL, they will be prompted to authorize your application for the requested scopes. On acceptance, they will be redirected to your `redirect_uri`, which will contain an additional querystring parameter, `code`. `state` will also be returned if previously sent, and should be validated at this point.

`prompt` controls how the authorization flow handles existing authorizations. If a user has previously authorized your application with the requested scopes and prompt is set to `consent`, it will request them to reapprove their authorization. If set to `none`, it will skip the authorization screen and redirect them back to your redirect URI without requesting their authorization. For passthrough scopes, like `bot` and `webhook.incoming`, authorization is always required.

The `integration_type` parameter specifies the [installation context](/developers/en-us/resources/application#installation-context) for the authorization. The installation context determines where the application will be installed, and is only relevant when `scope` contains `applications.commands`. When set to 0 (GUILD_INSTALL) the application will be authorized for installation to a server, and when set to 1 (USER_INSTALL) the application will be authorized for installation to a user. The application must be configured in the Developer Portal to support the provided `integration_type`.

## 授權碼憑證 (Authorization Code Grant)

授權碼憑證是大多數開發者所公認的「標準 OAuth2」，涉及獲取授權碼並將其更換為使用者的存取權杖 (access token)。它允許授權伺服器作為用戶端與資源擁有者之間的媒介，力求資源擁有者的認證憑證絕不會直接與用戶端共用。

所有對 OAuth2 端點的呼叫都需要 HTTP 基本驗證 (HTTP Basic authentication)，或者在表單資料主體中提供 `client_id` 與 `client_secret`。

<ManualAnchor id="authorization-code-grant-authorization-url-example" />
###### 授權網址範例 (Authorization URL Example)

```
https://discord.com/oauth2/authorize?response_type=code&client_id=157730590492196864&scope=identify%20guilds.join&state=15773059ghq9183habn&redirect_uri=https%3A%2F%2Fnicememe.website&prompt=consent&integration_type=0
```

`client_id` 是您應用程式的 `client_id`。`scope` 是由網址編碼空格 (`%20`) 分隔的 [OAuth2 權限範圍](/developers/bilingual/topics/oauth2#shared-resources-oauth2-scopes) 列表。`redirect_uri` 是您在建立應用程式時註冊的任何經網址編碼後的 URL。`state` 是 [狀態與安全性](/developers/bilingual/topics/oauth2#state-and-security) 中提到的唯一字串。

當有人導覽至此網址時，系統會提示他們為請求的權限範圍授權您的應用程式。接受後，他們將被重新導向至您的 `redirect_uri`，其中將包含一個額外的查詢字串參數 `code`。如果先前有發送 `state`，此時也會一併傳回，並應在此步驟進行驗證。

`prompt` 控制授權流程如何處理現有的授權。如果使用者先前已為您的應用程式授權了請求的權限範圍，且 prompt 設定為 `consent`，系統將要求他們重新批准其授權。如果設定為 `none`，它將跳過授權畫面，並將其重新導向回您的重新導向 URI，而不需要求他們授權。對於像 `bot` 和 `webhook.incoming` 這樣的傳遞 (passthrough) 權限範圍，則一律需要授權。

`integration_type` 參數指定了授權的 [安裝上下文 (installation context)](/developers/bilingual/resources/application#installation-context)。安裝上下文決定了應用程式的安裝位置，且僅在 `scope` 包含 `applications.commands` 時才相關。當設定為 0 (GUILD_INSTALL) 時，應用程式將被授權安裝到伺服器；當設定為 1 (USER_INSTALL) 時，應用程式將被授權安裝到使用者帳戶。應用程式必須在開發者入口網站中進行設定以支援所提供的 `integration_type`。

</div>

## Implicit Grant

The implicit OAuth2 grant is a simplified flow optimized for in-browser clients. Instead of issuing the client an authorization code to be exchanged for an access token, the client is directly issued an access token. The URL is formatted as follows:

<ManualAnchor id="implicit-grant-authorization-url-example" />
###### Authorization URL Example

```
https://discord.com/oauth2/authorize?response_type=token&client_id=290926444748734499&state=15773059ghq9183habn&scope=identify
```

On redirect, your redirect URI will contain additional **URI fragments**: `access_token`, `token_type`, `expires_in`, `scope`, and [`state`](/developers/en-us/topics/oauth2#state-and-security)(if specified). **These are not querystring parameters.** Be mindful of the "#" character:

<ManualAnchor id="implicit-grant-redirect-url-example" />
###### Redirect URL Example

```
https://findingfakeurlsisprettyhard.tv/#access_token=RTfP0OK99U3kbRtHOoKLmJbOn45PjL&token_type=Bearer&expires_in=604800&scope=identify&state=15773059ghq9183habn
```

There are tradeoffs in using the implicit grant flow. It is both quicker and easier to implement, but rather than exchanging a code and getting a token returned in a secure HTTP body, the access token is returned in the URI fragment, which makes it possibly exposed to unauthorized parties. **You also are not returned a refresh token, so the user must explicitly re-authorize once their token expires.**

Discord's API provides bot users, which are a separate type of user dedicated to automation. Bot users are automatically added to all apps, and are authenticated using the bot token found in your [app's settings](https://discord.com/developers/applications). Unlike the normal OAuth2 flow, bot users have full access to most API routes without using bearer tokens, and can connect to the [Real Time Gateway](/developers/en-us/events/gateway).

### Bot vs User Accounts

<Warning>
Developers must abide by the [terms of service](https://support-dev.discord.com/hc/articles/8562894815383-Discord-Developer-Terms-of-Service), which includes refraining from automating standard user accounts (generally called "self-bots") outside of the OAuth2/bot API.
</Warning>

Bot users have a few differences compared to standard Discord users:

1. Bots are added to guilds through the OAuth2 API, and cannot accept normal invites.
2. Bots cannot have friends or be added to or join Group DMs.
3. [Verified bots](https://support-dev.discord.com/hc/en-us/articles/23926564536471-How-Do-I-Get-My-App-Verified) do not have a maximum number of guilds.
4. Bots have an entirely separate set of [rate limits](/developers/en-us/topics/rate-limits#rate-limits).

### Bot Authorization Flow

Bot authorization is a special server-less and callback-less OAuth2 flow that makes it easy for users to add bots to guilds. The URL you create looks similar to what we use for full stack implementation:

<ManualAnchor id="bot-authorization-flow-bot-auth-parameters" />
###### Bot Auth Parameters

| name                  | description                                                                                                                          |
|-----------------------|--------------------------------------------------------------------------------------------------------------------------------------|
| client_id             | your app's client id                                                                                                                 |
| scope?                | needs to include `bot` for the bot flow                                                                                              |
| permissions?          | the [permissions](/developers/en-us/topics/permissions) you're requesting                                                                  |
| guild_id?             | pre-fills the dropdown picker with a guild for the user                                                                              |
| disable_guild_select? | `true` or `false`—disallows the user from changing the guild dropdown                                                                |
| integration_type?     | the [installation context](/developers/en-us/resources/application#application-object-application-integration-types) for the authorization |

<ManualAnchor id="bot-authorization-flow-url-example" />
###### URL Example

```
https://discord.com/oauth2/authorize?client_id=157730590492196864&scope=bot&permissions=1
```

In the case of bots, the `scope` parameter should be set to `bot`. There's also a new parameter, `permissions`, which is an integer corresponding to the [permission calculations](/developers/en-us/topics/permissions#permissions-bitwise-permission-flags) for the bot. You'll also notice the absence of `response_type` and `redirect_uri`. Bot authorization does not require these parameters because there is no need to retrieve the user's access token.

When the user navigates to this page, they'll be prompted to add the bot to a guild in which they have proper permissions. On acceptance, the bot will be added. Super easy!

If you happen to already know the ID of the guild the user will add your bot to, you can provide this ID in the URL as a `guild_id=GUILD_ID` parameter.
When the authorization page loads, that guild will be preselected in the dialog if that user has permission to add the bot to that guild. You can use this in conjunction with the parameter `disable_guild_select=true` to disallow the user from picking a different guild.

If you only provide the `client_id` parameter (with optionally `permissions`, `guild_id`, or `disable_guild_select`), the authorization will use the [default install settings](/developers/en-us/resources/application#configuring-an-install-link-and-default-install-settings) configured in the developer portal. Specifying `scope`, `integration_type`, or `redirect_uri` will override this behavior.

If your bot is super specific to your private clubhouse, or you just don't like sharing, you can leave the `Public Bot` option unchecked in your application's settings. If unchecked, only you can add the bot to guilds. If marked as public, anyone with your bot's URL can add it to guilds in which they have proper permissions.

### Bot 授權流程 (Bot Authorization Flow)

Bot 授權是一種特殊的無伺服器與無回呼的 OAuth2 流程，使使用者能輕鬆將 Bot 新增至伺服器。您建立的網址與我們用於全端實作的網址類似：

<ManualAnchor id="bot-authorization-flow-bot-auth-parameters" />
###### Bot 授權參數 (Bot Auth Parameters)

| 名稱 (Name)           | 說明 (Description)                                                                                                                          |
|-----------------------|--------------------------------------------------------------------------------------------------------------------------------------|
| client_id             | 您的應用程式用戶端 ID                                                                                                                 |
| scope?                | 對於 Bot 授權流程，需要包含 `bot` |
| permissions?          | 您所請求的 [權限 (permissions)](/developers/bilingual/topics/permissions)                                                                  |
| guild_id?             | 為使用者預先填入伺服器下拉式選單 |
| disable_guild_select? | `true` 或 `false` —— 禁止使用者變更下拉式選單中的伺服器選項 |
| integration_type?     | 授權的 [安裝上下文 (installation context)](/developers/bilingual/resources/application#application-object-application-integration-types) |

<ManualAnchor id="bot-authorization-flow-url-example" />
###### 網址範例 (URL Example)

```
https://discord.com/oauth2/authorize?client_id=157730590492196864&scope=bot&permissions=1
```

在 Bot 的情況下，`scope` 參數應設定為 `bot`。還有一個新參數 `permissions`，這是一個對應於 Bot 的 [權限計算](/developers/bilingual/topics/permissions#permissions-bitwise-permission-flags) 的整數。您還會注意到缺少 `response_type` 與 `redirect_uri`。Bot 授權不需要這些參數，因為無需檢索使用者的存取權杖。

當使用者導覽至此頁面時，系統會提示他們將 Bot 新增至他們擁有適當權限的伺服器。接受後，Bot 將被新增。超級簡單！

如果您已經知道使用者要將您的 Bot 新增到的伺服器 ID，您可以將此 ID 作為 `guild_id=GUILD_ID` 參數提供在網址中。當授權頁面載入時，如果該使用者擁有將 Bot 新增到該伺服器的權限，該伺服器將在對話框中預先選取。您可以將此參數與 `disable_guild_select=true` 結合使用，以禁止使用者挑選其他伺服器。

如果您僅提供 `client_id` 參數（可選填 `permissions`、`guild_id` 或 `disable_guild_select`），授權將使用在開發者入口網站中設定的 [預設安裝設定](/developers/bilingual/resources/application#configuring-an-install-link-and-default-install-settings)。指定 `scope`、`integration_type` 或 `redirect_uri` 將會覆寫此預設行為。

如果您的 Bot 非常特定於您的私人群組，或者您只是不喜歡與人分享，您可以在您應用程式的設定中取消勾選 `Public Bot` 選項。如果未勾選，則只有您可以將該 Bot 新增至伺服器。如果標記為公開，則任何擁有您 Bot 網址的人都可以將其新增至他們擁有適當權限的伺服器中。

## Webhooks

Discord's webhook flow is a specialized version of an [authorization code](/developers/en-us/topics/oauth2#authorization-code-grant) implementation. In this case, the `scope` querystring parameter needs to be set to `webhook.incoming`:

<ManualAnchor id="webhooks-url-example" />
###### URL Example

```
https://discord.com/oauth2/authorize?response_type=code&client_id=157730590492196864&scope=webhook.incoming&state=15773059ghq9183habn&redirect_uri=https%3A%2F%2Fnicememe.website
```

When the user navigates to this URL, they will be prompted to select a channel in which to allow the webhook. When the webhook is [executed](/developers/en-us/resources/webhook#execute-webhook), it will post its message into this channel. On acceptance, the user will be redirected to your `redirect_uri`. The URL will contain the `code` querystring parameter which should be [exchanged for an access token](/developers/en-us/topics/oauth2#authorization-code-grant-access-token-exchange-example). ## Get Current Authorization Information
<Route method="GET">/oauth2/@me</Route>

Returns info about the current authorization. Requires authentication with a bearer token.

<ManualAnchor id="get-current-authorization-information-response-structure" />
###### Response Structure

| Field       | Type                                                                               | Description                                                                       |
|-------------|------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------|
| application | partial [application](/developers/en-us/resources/application#application-object) object | the current application                                                           |
| scopes      | array of strings                                                                   | the scopes the user has authorized the application for                            |
| expires     | ISO8601 timestamp                                                                  | when the access token expires                                                     |
| user?       | [user](/developers/en-us/resources/user#user-object) object                              | the user who has authorized, if the user has authorized with the `identify` scope |

<ManualAnchor id="get-current-authorization-information-example-authorization-information" />
###### Example Authorization Information

```json
{
    "application": {

    },

        "guilds.join",

## 獲取當前授權資訊 (Get Current Authorization Information)
<Route method="GET">/oauth2/@me</Route>

傳回有關當前授權的資訊。需要使用 Bearer 權杖進行驗證。

<ManualAnchor id="get-current-authorization-information-response-structure" />
###### 回應結構 (Response Structure)

| 欄位 | 類型 | 說明 |
|-------------|------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------|
| application | partial [application](/developers/bilingual/resources/application#application-object) 物件 | 當前的應用程式 |
| scopes      | array of strings                                                                   | 使用者已授權給該應用程式的 Scope 權限範圍 |
| expires     | ISO8601 timestamp                                                                  | 存取權杖失效時間 |
| user?       | [user](/developers/bilingual/resources/user#user-object) 物件                              | 授權的使用者資訊，如果使用者是使用 `identify` 權限範圍授權 |

###### 授權資訊範例 (Example Authorization Information)

    },
    "scopes": [
        "guilds.join",
        "identify"
    ],
    "text": "..."
}
```
