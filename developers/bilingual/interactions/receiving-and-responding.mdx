An **[Interaction](/developers/en-us/interactions/receiving-and-responding#interaction-object)** is the message that your application receives when a user uses an application command or a message component.

For [Slash Commands](/developers/en-us/interactions/application-commands#slash-commands), it includes the values that the user submitted.

For [User Commands](/developers/en-us/interactions/application-commands#user-commands) and [Message Commands](/developers/en-us/interactions/application-commands#message-commands), it includes the resolved user or message on which the action was taken.

For [Message Components](/developers/en-us/components/reference) it includes identifying information about the component that was used. It will also include some metadata about how the interaction was triggered: the `guild_id`, `channel`, `member` and other fields. You can find all the values in our data models below.
</div>

一個 **[Interaction (互動)](/developers/bilingual/interactions/receiving-and-responding#interaction-object)** 是當使用者使用應用程式指令或訊息組件時，您的應用程式所接收到的訊息。

對於 [斜線指令 (Slash Commands)](/developers/bilingual/interactions/application-commands#slash-commands)，它包含使用者提交的數值。

對於 [使用者指令 (User Commands)](/developers/bilingual/interactions/application-commands#user-commands) 和 [訊息指令 (Message Commands)](/developers/bilingual/interactions/application-commands#message-commands)，它包含執行該操作的目標使用者或訊息物件。

對於 [訊息組件 (Message Components)](/developers/bilingual/components/reference) 它包含所使用組件的識別資訊。它還會包含一些有關如何觸發互動的中繼資料：`guild_id`、`channel`、`member` 以及其他欄位。您可以在下方的資料模型中找到所有的值。
</div>

###### Authorizing Integration Owners Object

The `authorizing_integration_owners` field includes details about the authorizing user or server for the installation(s) relevant to the interaction. For apps installed to a user, it can be used to tell the difference between the authorizing user and the user that triggered an interaction (like a message component).

A key will only be present if the following are true:
- The app has been authorized to the [installation context](/developers/en-us/resources/application#application-object-application-integration-types) corresponding to the key (`GUILD_INSTALL` or `USER_INSTALL`)
- The interaction is supported in the source [interaction context](/developers/en-us/interactions/receiving-and-responding#interaction-object-interaction-context-types) (`GUILD`, `BOT_DM`, or `PRIVATE_CHANNEL`) for the installation context corresponding to the key
- And for command invocations, the command must be supported in the installation context (using [`integration_types`](/developers/en-us/interactions/application-commands#contexts))

The values in `authorizing_integration_owners` depend on the key—

- If the key is `GUILD_INSTALL` (`"0"`), the value depends on the source of the interaction:
    - The value will be the guild ID if the interaction is triggered from a server
    - The value will be `"0"` if the interaction is triggered from a DM with the app's bot user
- If the key is `USER_INSTALL` (`"1"`), the value will be the ID of the authorizing user
</div>

###### 授權整合服務擁有者物件 (Authorizing Integration Owners Object)

`authorizing_integration_owners` 欄位包含與互動相關之安裝授權使用者或伺服器的詳細資訊。對於安裝到使用者的應用程式，它可用於區分授權使用者與觸發互動的使用者（例如點擊訊息組件的使用者）。

僅在以下條件成立時，Key 值才會存在：
- 應用程式已被授權到與該 Key 值對應的[安裝上下文](/developers/bilingual/resources/application#application-object-application-integration-types)（`GUILD_INSTALL` 或 `USER_INSTALL`）
- 該安裝上下文支援該互動在來源的 [互動上下文](/developers/bilingual/interactions/receiving-and-responding#interaction-object-interaction-context-types)（`GUILD`、`BOT_DM` 或 `PRIVATE_CHANNEL`）中運作
- 對於指令觸發，該指令必須在安裝上下文中受支援（使用 [`integration_types`](/developers/bilingual/interactions/application-commands#contexts)）

`authorizing_integration_owners` 中的值取決於 Key ——

- 如果 Key 是 `GUILD_INSTALL` (`"0"`)，其值取決於互動的來源：
    - 如果互動是在伺服器中觸發的，該值將是該伺服器的 ID
    - 如果互動是在與應用程式 Bot 使用者的私訊中觸發的，該值將是 `"0"`
- 如果 Key 是 `USER_INSTALL` (`"1"`)，該值將是授權使用者的 ID

## Receiving an Interaction

When a user interacts with your app, your app will receive an **[Interaction](/developers/en-us/interactions/receiving-and-responding#interaction-object)**. Your app can receive an interaction in one of two ways:

-   Via [Interaction Create](/developers/en-us/events/gateway-events#interaction-create) gateway event
-   Via outgoing webhook

These two methods are **mutually exclusive**; you can _only_ receive Interactions one of the two ways. The `INTERACTION_CREATE` [Gateway Event](/developers/en-us/events/gateway-events#interaction-create) may be handled by connected clients, while the webhook method detailed below does not require a connected client.

If you want to receive interactions via HTTP-based outgoing webhooks, you must configure an Interactions Endpoint URL for your app. You can read about preparing and adding an Interactions Endpoint URL to your app in the [Preparing for Interactions](/developers/en-us/interactions/overview#preparing-for-interactions) section in Interactions Overview.

### Interaction Metadata

An [Interaction](/developers/en-us/interactions/receiving-and-responding#interaction-object) includes metadata to aid your application in handling it as well as `data` specific to the interaction type. You can find samples for each interaction type on their respective pages:

-   [Slash Commands](/developers/en-us/interactions/application-commands#slash-commands-example-interaction)
-   [User Commands](/developers/en-us/interactions/application-commands#user-commands-example-interaction)
-   [Message Commands](/developers/en-us/interactions/application-commands#message-commands-example-interaction)
-   [Message Components](/developers/en-us/components/using-message-components)
-   [Modal Components](/developers/en-us/components/using-modal-components)

An explanation of all the fields can be found in our [data models](/developers/en-us/interactions/receiving-and-responding#interaction-object).

Now that you've gotten the data from the user, it's time to respond to them.

## 接收 Interaction (Receiving an Interaction)

當使用者與您的應用程式互動時，您的應用程式將會收到一個 **[Interaction](/developers/bilingual/interactions/receiving-and-responding#interaction-object)**。您的應用程式可以透過以下兩種方式之一來接收互動：

-   透過 [Interaction Create](/developers/bilingual/events/gateway-events#interaction-create) 閘道器事件
-   透過連出 Webhook

這兩種方法是**互斥的**；您*只能*透過其中一種方式接收互動。連接的用戶端可以處理 `INTERACTION_CREATE` [閘道器事件](/developers/bilingual/events/gateway-events#interaction-create)，而下方詳述的 Webhook 方法則不需要連接的用戶端。

如果您希望透過基於 HTTP 的連出 Webhooks 接收互動，您必須為您的應用程式配置互動端點網址。您可以在互動總覽的[為互動做準備](/developers/bilingual/interactions/overview#preparing-for-interactions)區段中閱讀有關為您的應用程式準備和新增互動端點網址的資訊。

所有欄位的說明都可以在我們的[資料模型](/developers/bilingual/interactions/receiving-and-responding#interaction-object)中找到。

## Responding to an Interaction

Interactions--both receiving and responding--are webhooks under the hood. So responding to an Interaction is just like sending a webhook request!

<Info>
Interaction responses have the same header requirements as normal HTTP API requests. See [here](/developers/en-us/reference#http-api) for further information.
</Info>

There are a number of ways you can respond to an interaction:

## Interaction Callback

When responding to an interaction received, you can make a `POST` request to `/interactions/<interaction_id>/<interaction_token>/callback`. `interaction_id` is the unique id of that individual Interaction from the received payload. `interaction_token` is the unique token for that interaction from the received payload.

If you are receiving Interactions over the gateway, you **have to respond via HTTP**. Responses to Interactions **are not sent as commands over the gateway**.

**If you send this request for an interaction received over HTTP, respond to the original HTTP request with a 202 and no body.**

```py
import requests

url = "https://discord.com/api/v10/interactions/<interaction_id>/<interaction_token>/callback"

json = {
    "type": 4,
    "data": {
        "content": "Congrats on sending your command!"
    }
}
r = requests.post(url, json=json)
```

<Info>
Interaction `tokens` are valid for **15 minutes** and can be used to send followup messages but you **must send an initial response within 3 seconds of receiving the event**. If the 3 second deadline is exceeded, the token will be invalidated.
</Info>

<Accordion title="Inline HTTP Response Behavior" icon="code">
    If you receive interactions over HTTP, your server can also respond to the received `POST` request. You'll want to respond with a `200` status code (if everything went well), as well as specifying a `type` and `data`, which is an [Interaction Response](/developers/en-us/interactions/receiving-and-responding#interaction-response-object) object:

    ```py
    @app.route('/', methods=['POST'])
    def my_command():
        if request.json["type"] == 1:
            return jsonify({
                "type": 1
            })

        else:
            return jsonify({
                "type": 4,
                "data": {
                    "tts": False,
                    "content": "Congrats on sending your command!",
                    "embeds": [],
                    "allowed_mentions": { "parse": [] }
                }
            })
    ```
</Accordion>

## Interaction 回呼 (Interaction Callback)

當回覆收到的互動時，您可以對 `/interactions/<interaction_id>/<interaction_token>/callback` 發送 `POST` 請求。`interaction_id` 是接收到的 payload 中該特定 Interaction 的唯一 ID。`interaction_token` 是接收到的 payload 中該互動的唯一 token。

如果您是透過閘道器接收 Interaction，您**必須透過 HTTP 做出回覆**。對 Interaction 的回覆**不會作為命令透過閘道器發送**。

**如果您對透過 HTTP 接收到的互動發送此請求，請以 202 且無 body 回應原始 HTTP 請求。**

```py
import requests

url = "https://discord.com/api/v10/interactions/<interaction_id>/<interaction_token>/callback"

json = {
    "type": 4,
    "data": {
        "content": "Congrats on sending your command!"
    }
}
r = requests.post(url, json=json)
```

<Info>
互動 `tokens` 的有效期限為 **15 分鐘**，可用於發送後續訊息，但您**必須在收到事件後的 3 秒內發送初始回應**。如果超過 3 秒的截止時間，token 將會失效。
</Info>

<Accordion title="內聯 HTTP 回應行為" icon="code">

## Followup Messages

Sometimes, you want to send followup messages to a user after responding to an interaction. Or, you may want to edit your original response. Whether you receive Interactions over the gateway or by outgoing webhook, you can use the following endpoints to edit your initial response or send followup messages:

-   [`PATCH /webhooks/<application_id>/<interaction_token>/messages/@original`](/developers/en-us/interactions/receiving-and-responding#edit-original-interaction-response) to edit your initial response to an Interaction
-   [`DELETE /webhooks/<application_id>/<interaction_token>/messages/@original`](/developers/en-us/interactions/receiving-and-responding#delete-original-interaction-response) to delete your initial response to an Interaction
-   [`POST /webhooks/<application_id>/<interaction_token>`](/developers/en-us/interactions/receiving-and-responding#create-followup-message) to send a new followup message
-   [`PATCH /webhooks/<application_id>/<interaction_token>/messages/<message_id>`](/developers/en-us/interactions/receiving-and-responding#edit-followup-message) to edit a message sent with that `token`

<Info>

## Create Followup Message
<Route method="POST">/webhooks/[\{application.id\}](/developers/en-us/resources/application#application-object)/[\{interaction.token\}](/developers/en-us/interactions/receiving-and-responding#interaction-object)</Route>

<Info>
Apps are limited to 5 followup messages per interaction if it was initiated from a user-installed app and isn't installed in the server (meaning the [authorizing integration owners object](/developers/en-us/interactions/receiving-and-responding#interaction-object-authorizing-integration-owners-object) only contains `USER_INSTALL`)
</Info>

Create a followup message for an Interaction. Functions the same as [Execute Webhook](/developers/en-us/resources/webhook#execute-webhook), but `wait` is always true. The `thread_id`, `avatar_url`, and `username` parameters are not supported when using this endpoint for interaction followups. You can use the `EPHEMERAL` [message flag](/developers/en-us/resources/message#message-object-message-flags) `1 << 6` (64) to send a message that only the user can see. You can also use the `IS_COMPONENTS_V2` [message flag](/developers/en-us/resources/message#message-object-message-flags) `1 << 15` (32768) to send a [component](/developers/en-us/components/reference)-based message.

When using this endpoint directly after responding to an interaction with `DEFERRED_CHANNEL_MESSAGE_WITH_SOURCE`, this endpoint will function as [Edit Original Interaction Response](/developers/en-us/interactions/receiving-and-responding#edit-original-interaction-response) for backwards compatibility. In this case, no new message will be created, and the loading message will be edited instead. The ephemeral flag will be ignored, and the value you provided in the initial defer response will be preserved, as an existing message's ephemeral state cannot be changed. This behavior is deprecated, and you should use the [Edit Original Interaction Response](/developers/en-us/interactions/receiving-and-responding#edit-original-interaction-response) endpoint in this case instead.
</div>

## 建立後續訊息 (Create Followup Message)
<Route method="POST">/webhooks/[\{application.id\}](/developers/bilingual/resources/application#application-object)/[\{interaction.token\}](/developers/bilingual/interactions/receiving-and-responding#interaction-object)</Route>

<Info>
如果互動是從使用者安裝型 App 發起的，且未安裝在伺服器中（代表[授權整合服務擁有者物件](/developers/bilingual/interactions/receiving-and-responding#interaction-object-authorizing-integration-owners-object)僅包含 `USER_INSTALL`），則每次互動最多只能發送 5 條後續訊息。
</Info>

為 Interaction 建立後續訊息。功能與[執行 Webhook](/developers/bilingual/resources/webhook#execute-webhook)相同，但 `wait` 一律為 true。在使用此端點進行互動後續處理時，不支援 `thread_id`、`avatar_url` 和 `username` 參數。您可以使用 `EPHEMERAL` [訊息旗標](/developers/bilingual/resources/message#message-object-message-flags) `1 << 6` (64) 來發送僅使用者可見的訊息。您也可以使用 `IS_COMPONENTS_V2` [訊息旗標](/developers/bilingual/resources/message#message-object-message-flags) `1 << 15` (32768) 來發送基於[組件](/developers/bilingual/components/reference)-based 訊息。

在使用 `DEFERRED_CHANNEL_MESSAGE_WITH_SOURCE` 回應互動後立即使用此端點時，此端點將發揮與[編輯原始互動回應](/developers/bilingual/interactions/receiving-and-responding#edit-original-interaction-response)相同的功能以實現向後相容。在此情況下，不會建立新訊息，而是會編輯載入訊息。臨時 (ephemeral) 旗標將被忽略，並且您在初始延遲回應中提供的值將被保留，因為現有訊息的臨時狀態無法更改。此行為已被棄用，在此情況下您應改用[編輯原始互動回應](/developers/bilingual/interactions/receiving-and-responding#edit-original-interaction-response)端點。
