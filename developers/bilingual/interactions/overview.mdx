## Types of Interactions

There are different types of interactions in your app's toolbelt that can pick and choose to build engaging, interactive experiences in Discord.

### Commands

**[Application commands](/developers/en-us/interactions/application-commands)** provide users a native way to invoke an app in Discord. They often map to an app's core features or functionality.

![Command launcher in the Desktop client](/images/overview-command-desktop.png)

When an app creates a command it can choose the command's type, which determines where it appears in the Discord client and the metadata the app will receive when the command is invoked. There are three types for application commands:

- **Slash commands** are the most common type of command and are accessed by typing `/` in the chat input, or by opening the command picker.
- **Message commands** are commands related to a message or a message's content. They're accessed by clicking on the context menu (the three dots) at the top-right of a message (or right clicking on a message), then navigating to the "Apps" section.
- **User commands** are commands that related to a user in Discord. They're accessed by right clicking on a user profile, then navigating to the "Apps" section.
- **Entry Point commands** are commands used as the primary way to launch [Activities](/developers/en-us/activities/overview) from the App Launcher.

Details about creating commands and handling command interactions are in the [Application Commands](/developers/en-us/interactions/application-commands) documentation.

## 互動類型 (Types of Interactions)

在您應用程式的工具箱中，有不同類型的互動功能可供挑選，以在 Discord 中建置吸引人且具備互動性的體驗。

### 指令 (Commands)

**[應用程式指令 (Application commands)](/developers/bilingual/interactions/application-commands)** 為使用者提供在 Discord 中原生啟動應用程式的方法。它們通常對應於應用程式的核心特徵或功能。

![電腦版用戶端中的指令啟動器](/images/overview-command-desktop.png)

當應用程式建立指令時，可以選擇指令的類型，這決定了它在 Discord 用戶端中顯示的位置，以及在啟動該指令時應用程式會收到的中繼資料。應用程式指令有三種類型：

- **斜線指令 (Slash commands)** 是最常見的指令類型，可藉由在聊天輸入框中輸入 `/` 或開啟指令選取器來存取。
- **訊息指令 (Message commands)** 是與訊息或訊息內容相關的指令。它們可以透過點擊訊息右上角的快顯功能表（三個點，或右鍵點擊訊息），然後導覽至「應用程式 (Apps)」區段來存取。
- **使用者指令 (User commands)** 是與 Discord 使用者相關的指令。它們可以透過右鍵點擊使用者個人資料，然後導覽至「應用程式 (Apps)」區段來存取。
- **進入點指令 (Entry Point commands)** 是使用者從應用程式啟動器 (App Launcher) 啟動 [互動活動 (Activities)](/developers/bilingual/activities/overview) 的主要方式。

有關建立指令與處理指令互動的詳細資訊，請參閱 [應用程式指令 (Application Commands)](/developers/bilingual/interactions/application-commands) 說明文件。

### Message Components

**[Message components](/developers/en-us/components/reference)** are interactive elements that can be included in the content of a message that your app sends in Discord. 

![Button message components in a message](/images/overview-components.png)

The main interactive components that apps can send in messages include:

- [Buttons](/developers/en-us/components/reference#button) are clickable components that can be customized with different styles, texts, and emoji.
- [Static select menus](/developers/en-us/components/reference#string-select) are components that a user can open to see a list of developer-defined select options with custom labels and descriptions.
- [Auto-populated select menus](/developers/en-us/components/reference#string-select) are a set of four different select components that are populated with contextual Discord resources, like a list of users or channels in a server.

A list of all message components and details on sending and receiving component interactions is in the [Message Components](/developers/en-us/components/reference) documentation.

### Modals

**[Modals](/developers/en-us/interactions/receiving-and-responding#interaction-response-object-modal)** are single-user pop-up interfaces that allow apps to collect form-like data. Modals can only be opened in response to a user invoking one of your app's commands or message components.

![Modals in the Discord client](/images/overview-modals.png)

The only interactive component that modals can contain are [text inputs](/developers/en-us/components/reference#text-input), which allow users to fill out single-or-multi line form inputs.

Details about creating and using modals is in the [Receiving and Responding](/developers/en-us/interactions/receiving-and-responding#interaction-response-object-modal) documentation.

## Preparing for Interactions

When a user interacts with your app, you have the option for your app to receive interactions in two mutually-exclusive ways:
- WebSocket-based Gateway connection
- HTTP via outgoing webhooks

By default your app will receive interactions via a Gateway connection, but you can opt-in to HTTP-based interactions by adding a **Interactions Endpoint URL** to your app's settings. Technical details about handling interactions is in the [Receiving and Responding](/developers/en-us/interactions/receiving-and-responding) documentation.

### Configuring an Interactions Endpoint URL

A **Interactions Endpoint URL** is a public endpoint for your app where Discord can send your app HTTP-based interactions. If your app is using [Gateway](/developers/en-us/events/gateway)-based interactions, you don't need to configure an Interactions Endpoint URL.

## 為互動功能做準備 (Preparing for Interactions)

當使用者與您的應用程式進行互動時，您可以選擇透過以下兩種互斥方式之一來接收互動：
- 基於 WebSocket 的閘道器 (Gateway) 連線
- 透過連出 Webhooks 的 HTTP 協定

預設情況下，您的應用程式將透過閘道器連線接收互動，但您可以透過在應用程式設定中新增 **Interactions Endpoint URL (互動端點網址)** 來選擇使用基於 HTTP 的互動。處理互動的技術細節請參閱 [接收與回覆 (Receiving and Responding)](/developers/bilingual/interactions/receiving-and-responding) 說明文件。

### 配置互動端點網址 (Configuring an Interactions Endpoint URL)

**互動端點網址 (Interactions Endpoint URL)** 是您應用程式的公開端點，Discord 可以向其發送基於 HTTP 的互動請求。如果您的應用程式使用基於 [閘道器 (Gateway)](/developers/bilingual/events/gateway) 的互動，則不需要配置互動端點網址。

#### Setting Up an Endpoint

Before you can add your Interactions Endpoint URL to your app, your endpoint must be prepared for two things ahead of time:

1. Acknowledging `PING` requests from Discord
2. Validate security-related request headers (`X-Signature-Ed25519` and `X-Signature-Timestamp`)

If either of these are not complete, your Interactions Endpoint URL will not be validated. Details on acknowledging PING requests and validating security-related headers are in the sections below.

#### Acknowledging PING requests

When adding your Interactions Endpoint URL, Discord will send a `POST` request with a `PING` payload with a `type: 1` to your endpoint. Your app is expected to acknowledge the request by returning a `200` response with a `PONG` payload (which has the same `type: 1`). Details about interaction responses are in the [Receiving and Responding documentation](/developers/en-us/interactions/receiving-and-responding).

<Info>
You must provide a valid `Content-Type` when responding to `PING`s. See [here](/developers/en-us/reference#http-api) for further information.
</Info>

<Accordion title="Responding to PING Requests" description="Code example for acknowledging PING interactions" icon="code">
To properly acknowledge a `PING` payload, return a `200` response with a payload of `type: 1`:

```py
@app.route('/', methods=['POST'])
def my_command():
    if request.json["type"] == 1:
        return jsonify({
            "type": 1
        })
```
</Accordion>

###### Validating Security Request Headers

The internet is a scary place, especially for people hosting public, unauthenticated endpoints. To receive interactions via HTTP, there are some security steps you **must** take before your app is eligible to receive requests.

Each interaction is sent with the following headers:

-   `X-Signature-Ed25519` as a signature
-   `X-Signature-Timestamp` as a timestamp

Using your favorite security library, you **must validate the request each time you receive an [interaction](/developers/en-us/interactions/receiving-and-responding#interaction-object)**. If the signature fails validation, your app should respond with a `401` error code.

<Accordion title="Validating Security Headers" description="Code example for validating security-related request headers" icon="code">
Below are some code examples that show how to validate the headers sent in interactions requests.

**JavaScript**

```js
const nacl = require("tweetnacl");

// Your public key can be found on your application in the Developer Portal
const PUBLIC_KEY = "APPLICATION_PUBLIC_KEY";

const signature = req.get("X-Signature-Ed25519");
const timestamp = req.get("X-Signature-Timestamp");
const body = req.rawBody; // rawBody is expected to be a string, not raw bytes

const isVerified = nacl.sign.detached.verify(
    Buffer.from(timestamp + body),
    Buffer.from(signature, "hex"),
    Buffer.from(PUBLIC_KEY, "hex")
);

if (!isVerified) {
    return res.status(401).end("invalid request signature");
}
```

**Python**

```py
from nacl.signing import VerifyKey
from nacl.exceptions import BadSignatureError

# Your public key can be found on your application in the Developer Portal
PUBLIC_KEY = 'APPLICATION_PUBLIC_KEY'

verify_key = VerifyKey(bytes.fromhex(PUBLIC_KEY))

signature = request.headers["X-Signature-Ed25519"]
timestamp = request.headers["X-Signature-Timestamp"]
body = request.data.decode("utf-8")

try:
    verify_key.verify(f'{timestamp}{body}'.encode(), bytes.fromhex(signature))
except BadSignatureError:
    abort(401, 'invalid request signature')
```
</Accordion>

In addition to ensuring your app validates security-related request headers at the time of saving your endpoint, Discord will also perform automated, routine security checks against your endpoint, including purposefully sending you invalid signatures. If you fail the validation, we will remove your interactions URL and alert you via email and System DM.

We highly recommend checking out our [Community Resources](/developers/en-us/developer-tools/community-resources#interactions) and the libraries found there. They not only provide typing for Interactions data models, but also include decorators for API frameworks like Flask and Express to make validation easy.

###### 驗證安全請求標頭 (Validating Security Request Headers)

網路世界是個危險的地方，特別是對託管公開、未經驗證之端點的開發者而言。若要透過 HTTP 接收互動，在您的應用程式有資格接收請求之前，您**必須**採取一些安全性步驟。

每個互動請求都會隨附以下標頭發送：

-   以 `X-Signature-Ed25519` 作為簽章 (signature)
-   以 `X-Signature-Timestamp` 作為時間戳記 (timestamp)

使用您最喜愛的安全性函式庫，**每次收到 [互動 (interaction)](/developers/bilingual/interactions/receiving-and-responding#interaction-object) 時，您都必須驗證該請求**。如果簽章驗證失敗，您的應用程式應回應 `401` 錯誤碼。

<Accordion title="驗證安全標頭" description="驗證安全相關請求標頭的程式碼範例" icon="code">
以下是一些程式碼範例，示範如何驗證互動請求中發送的標頭。

**JavaScript**

```js
const nacl = require("tweetnacl");

// 您的公鑰可在開發者入口網站的應用程式設定中找到
const PUBLIC_KEY = "APPLICATION_PUBLIC_KEY";

const signature = req.get("X-Signature-Ed25519");
const timestamp = req.get("X-Signature-Timestamp");
const body = req.rawBody; // rawBody 預期應為字串，而非原始位元組

const isVerified = nacl.sign.detached.verify(
    Buffer.from(timestamp + body),
    Buffer.from(signature, "hex"),
    Buffer.from(PUBLIC_KEY, "hex")
);

if (!isVerified) {
    return res.status(401).end("invalid request signature");
}
```

**Python**

```py
from nacl.signing import VerifyKey
from nacl.exceptions import BadSignatureError

# 您的公鑰可在開發者入口網站的應用程式設定中找到
PUBLIC_KEY = 'APPLICATION_PUBLIC_KEY'

verify_key = VerifyKey(bytes.fromhex(PUBLIC_KEY))

signature = request.headers["X-Signature-Ed25519"]
timestamp = request.headers["X-Signature-Timestamp"]
body = request.data.decode("utf-8")

try:
    verify_key.verify(f'{timestamp}{body}'.encode(), bytes.fromhex(signature))
except BadSignatureError:
    abort(401, 'invalid request signature')
```
</Accordion>

除了在儲存端點時確保您的應用程式驗證安全相關的請求標頭外，Discord 也會對您的端點執行自動化、例行性的安全性檢查，包括故意發送無效的簽章給您。如果您驗證失敗，我們將移除您的互動網址，並透過電子郵件和系統私訊通知您。

我們強烈建議查看我們的 [社群資源](/developers/bilingual/developer-tools/community-resources#interactions) 以及在其中找到的函式庫。它們不僅為互動資料模型提供類型定義，還包括適用於 Flask 和 Express 等 API 框架的裝飾器 (decorators)，以使驗證變取更加容易。

#### 新增互動端點網址 (Adding an Interactions Endpoint URL)

在您擁有可用作應用程式互動端點網址的公開端點後，您可以前往 [應用程式設定](https://discord.com/developers/applications) 將其新增至應用程式中。

在 [**General Information** 頁面](https://discord.com/developers/applications/select/information) 上，尋找 **Interactive Endpoint URL** 欄位。貼上您已設定為確認 `PING` 訊息且正確處理安全相關簽章標頭的公開網址。

## 處理互動功能 (Handling Interactions)

一旦您的應用程式為互動做好了準備，您可以探索 [接收與回覆 (Receiving and Responding)](/developers/bilingual/interactions/receiving-and-responding) 說明文件，該文件深入探討了在您應用程式中處理互動請求的技術細節。
