權限是限制並授予 Discord 使用者特定能力的一種方式。您可以在伺服器 (guild) 層級為不同的身分組 (roles) 設定一組基本權限。當這些身分組指派給使用者時，它們會授予或撤銷使用者在伺服器中的特定特權。除了伺服器層級的權限外，Discord 還支援針對單一頻道指派給特定身分組或成員的頻道覆寫權限 (permission overwrites)。

<Info>
[應用程式指令權限 (Application command permissions)](/developers/zh-tw/interactions/application-commands#permissions) 允許您除了設定特定身分組或使用者之外，還能為整個頻道啟用或停用特定指令。
</Info>

權限會儲存在序列化為字串的可變長度整數中，並使用位元運算 (bitwise operations) 進行計算。例如，權限值 `123` 會被序列化為 `"123"`。為了長期的穩定性，建議使用您偏好的程式語言的「大整數 (Big Integer)」函式庫來還原序列化權限。總權限整數可以藉由將每個獨立值進行按位或 (`|`, OR) 運算來確定，而標記 (flags) 則可以使用按位與 (`&`, AND) 運算來檢查。

在 API v8 及以上版本中，所有的權限都會序列化為字串，包括覆寫權限中的 `allow` 和 `deny` 欄位。任何新增的權限都會合併回基本欄位中。

<Info>
在 [API v6（現已棄用）](/developers/reference) 中，身分組和覆寫權限中的 `permissions`、`allow` 和 `deny` 欄位仍然序列化為數字；然而，這些數字不得超過 31 位元。在 API v6 的剩餘生命週期中，所有新的權限位元都將僅在 `permissions_new`、`allow_new` 和 `deny_new` 中引入。這些 `_new` 欄位僅用於回應序列化；使用這些欄位的請求應繼續使用接受字串或數字值的原始 `permissions`、`allow` 和 `deny` 欄位。
</Info>

```python
# 可傳送訊息 (Send Messages, 0x800) 與新增反應 (Add Reactions, 0x40) 的權限值：
permissions = 0x40 | 0x800 # 2112

# 檢查是否有設定對應的標記 (flags)：
(permissions & 0x40) == 0x40   # True (正確)
(permissions & 0x800) == 0x800 # True (正確)

# 剔除成員 (Kick Members, 0x2) 尚未被設定：
(permissions & 0x2) == 0x2 # False (錯誤)
```

當涉及頻道覆寫權限時，需要額外的邏輯；這將在下文進一步解釋。有關位元運算和標記的更多資訊，請參閱[此頁面](https://en.wikipedia.org/wiki/Bit_field)。

## 權限階層 (Permission Hierarchy)

權限的套用方式起初看起來很直觀，但也存在一些隱藏的限制，用以防止 Bot 根據其最高身分組與目標最高身分組的比較來執行某些不當行為。Bot 或使用者的最高身分組是指該身分組在伺服器中擁有最大的位置值，預設的 @everyone 身分組從 0 開始。權限遵循以下階層規則：

- Bot 可以為其他使用者指派位置低於自己最高身分組的身分組。
- Bot 可以編輯位置低於自己最高身分組的身分組，但只能為這些身分組授予自己擁有的權限。
- Bot 只能排序低於自己最高身分組的身分組。
- Bot 只能對其最高身分組低於 Bot 最高身分組的使用者執行剔除、封鎖以及編輯暱稱的操作。

除此之外，權限並不服從身分組階層。例如，使用者擁有兩個身分組：A 和 B。A 在 #coolstuff 頻道上拒絕了 `VIEW_CHANNEL` 權限。B 在同一個 #coolstuff 頻道上允許了 `VIEW_CHANNEL` 權限。不論身分組位置如何，使用者最終都能檢視 #coolstuff 頻道。

## 頻道覆寫權限 (Permission Overwrites)

覆寫權限可用於在頻道層級將特定權限套用到身分組或成員。適用的權限在上述表格中以 **T**（文字頻道）、**V**（語音頻道）或 **S**（講台頻道）標記。

當使用覆寫權限時，使用者在某些情況下可能會遇到權限衝突；也就是說，使用者可能擁有一些相互矛盾的覆寫權限，或是與其伺服器層級身分組權限相衝突。考慮到這一點，套用到使用者的權限具有以下優先順序階層：

1. 伺服器層級上套用給 @everyone 的基本權限
2. 伺服器層級上由使用者的身分組所允許的權限
3. 頻道層級上為 @everyone 拒絕的覆寫權限
4. 頻道層級上為 @everyone 允許的覆寫權限
5. 頻道層級上為特定身分組拒絕的覆寫權限
6. 頻道層級上為特定身分組允許的覆寫權限
7. 頻道層級上為特定成員拒絕的成員專屬覆寫權限
8. 頻道層級上為特定成員允許的成員專屬覆寫權限

以下虛擬碼以程式化方式示範了此過程：

```python
def compute_base_permissions(member, guild):
    if guild.is_owner(member):
        return ALL

    role_everyone = guild.get_role(guild.id)  # 獲取 @everyone 身分組
    permissions = role_everyone.permissions

    for role in member.roles:
        permissions |= role.permissions

    if permissions & ADMINISTRATOR == ADMINISTRATOR:
        return ALL

    return permissions

def compute_overwrites(base_permissions, member, channel):
    # ADMINISTRATOR 繞過所有潛在的頻道覆寫權限，因此在此處無需做任何處理。
    if base_permissions & ADMINISTRATOR == ADMINISTRATOR:
        return ALL

    permissions = base_permissions
    overwrite_everyone = overwrites.get(channel.guild_id)  # 尋找 (@everyone) 身分組覆寫並套用它。
    if overwrite_everyone:
        permissions &= ~overwrite_everyone.deny
        permissions |= overwrite_everyone.allow

    # 套用特定身分組覆寫。
    overwrites = channel.permission_overwrites
    allow = NONE
    deny = NONE
    for role_id in member.roles:
        overwrite_role = overwrites.get(role_id)
        if overwrite_role:
            allow |= overwrite_role.allow
            deny |= overwrite_role.deny

    permissions &= ~deny
    permissions |= allow

    # 套用成員專屬覆寫（如果存在）。
    overwrite_member = overwrites.get(member.user_id)
    if overwrite_member:
        permissions &= ~overwrite_member.deny
        permissions |= overwrite_member.allow

    return permissions

def compute_permissions(member, channel):
    base_permissions = compute_base_permissions(member, channel.guild)
    return compute_overwrites(base_permissions, member, channel)
```

## 隱含權限 (Implicit Permissions)

在 Discord 中，權限有時會根據邏輯使用隱含地被拒絕或允許。兩種最常見的情況是文字頻道的 `VIEW_CHANNEL` 和 `SEND_MESSAGES`。在某個頻道上為使用者或身分組拒絕 `VIEW_CHANNEL` 會隱含地拒絕該頻道上的其他所有權限。儘管像 `SEND_MESSAGES` 這樣的權限並未明確為使用者拒絕，但它們會被忽略，因為使用者無法檢視或讀取該頻道的訊息。

拒絕 `SEND_MESSAGES` 會隱含地拒絕 `MENTION_EVERYONE`、`SEND_TTS_MESSAGES`、`ATTACH_FILES` 和 `EMBED_LINKS`。同樣地，這些權限在進行權限計算時並未明確被拒絕，但它們會被忽略，因為使用者無法執行傳送訊息的基本動作。

對於語音和講台頻道，拒絕 `CONNECT` 權限也會隱含地拒絕其他權限，例如 `MANAGE_CHANNEL`。

在其他情況下，某些權限也可能會隱含地拒絕或允許其他權限。在所有情況下，這都是基於對擁有特定權限的使用者應如何或不應如何與 Discord 互動的邏輯推論。

## 繼承權限，貼文/執行緒 (Inherited Permissions)

貼文/執行緒會繼承父頻道（其建立所在的頻道）的權限，但有一個例外：`SEND_MESSAGES` 權限不會被繼承；使用者必須擁有 `SEND_MESSAGES_IN_THREADS` 權限才能在貼文內傳送訊息，這允許使用者在公告頻道等地方參與討論。

使用者必須擁有 `VIEW_CHANNEL` 權限才能檢視頻道中的_所有_貼文，即使他們被直接提及或加入到該貼文中也是如此。

## 權限同步 (Permission Syncing)

關於類別 (categories) 以及類別中頻道的權限運作會有些微妙。這並非單純的繼承，而是透過我們稱為「權限同步 (Permission Syncing)」的方式計算。如果子頻道具有與其父類別相同的權限和覆寫權限（或兩者皆無），則該頻道被視為與該類別「同步」。對**父類別**進行的任何進一步變更都會反映在其已同步的子頻道中。對**子頻道**進行的任何進一步變更都會導致其與父類別「去同步 (de-synced)」，其權限將不再隨父類別的變更而變動。

### Role 物件 (Role Object)

身分組 (Roles) 代表附加到一組使用者的權限集合。身分組有名稱、顏色，且可以「固定 (hoist)」在成員列表側邊欄中，使其成員被單獨列出。身分組可以針對全域上下文 (guild) 和頻道上下文配置獨立的權限設定。`@everyone` 身分組的 ID 與其所屬的伺服器 ID 相同。

| managed        | boolean                                                                             | 此身分組是否由整合服務所管理 |
| mentionable    | boolean                                                                             | 此身分組是否可被提及 |
| tags?          | [身分組標籤](/developers/zh-tw/topics/permissions#role-object-role-tags-structure) 物件  | 此身分組擁有的標籤 |
| flags          | integer                                                                             | 身分組 [旗標](/developers/zh-tw/topics/permissions#role-object-role-flags)，組合成 [位元欄位](https://en.wikipedia.org/wiki/Bit_field) |

沒有顏色的身分組 (`colors.primary_color == 0`) 不會計入使用者列表中最終計算出的顏色。

\* `color` 仍會被 API 傳回，但建議在發送請求時使用 `colors` 欄位。

<ManualAnchor id="role-object-role-tags-structure" />
###### 身分組標籤結構 (Role Tags Structure)

類型為 `null` 的標籤代表布林值。如果為「真 (true)」，則它們將存在並設定為 `null`，如果為「假 (false)」，則它們不會存在。

| 欄位 | 類型 | 說明 |
|--------------------------|-----------|----------------------------------------------------|
| bot_id?                  | snowflake | 此身分組所屬的 Bot ID |
| integration_id?          | snowflake | 此身分組所屬的整合服務 ID |
| premium_subscriber?      | null      | 此身分組是否為伺服器的加成者 (Booster) 身分組 |
| subscription_listing_id? | snowflake | 此身分組訂閱 SKU 和品項的 ID |
| available_for_purchase?  | null      | 此身分組是否可供購買 |
| guild_connections?       | null      | 此身分組是否為伺服器的連結身分組 |

<ManualAnchor id="role-object-role-colors-object" />
###### 身分組顏色物件 (Role Colors Object)

此物件一律會填入身分組的 `color` 作為 `primary_color`。只有在伺服器具有 `ENHANCED_ROLE_COLORS` [伺服器特徵 (guild feature)](/developers/zh-tw/resources/guild#guild-object-guild-features) 時，其他欄位才能設定為非 null 值。

| 欄位 | 類型 | 說明 |
|-----------------|----------|--------------------------------------------------------------------------------------------------------|
| primary_color   | integer  | 身分組的主要顏色 |
| secondary_color | ?integer | 身分組的次要顏色，這將使身分組呈現所提供顏色之間的漸層色 |
| tertiary_color  | ?integer | 身分組的第三顏色，這將使漸層呈現全像投影風格 (holographic style) |

<Info>
發送 `tertiary_color` 時，API 會強制身分組顏色為特定全像風格數值：
`primary_color = 11127295`、`secondary_color = 16759788`、且 `tertiary_color = 16761760`。
</Info>

<ManualAnchor id="role-object-default-role-colors-object" />
###### 預設身分組顏色物件 (Default Role Colors Object)

```json
"colors": {
  "primary_color": 0,
  "secondary_color": null,
  "tertiary_color": null
}
```

###### 身分組旗標 (Role Flags)

| 旗標 | 數值 | 說明 |
|-----------|----------|----------------------------------------------------------------------------------------------------------------|
| IN_PROMPT | `1 << 0` | 身分組可在[新手引導過程 (onboarding)](/developers/zh-tw/resources/guild#guild-onboarding-object)中供成員選擇 |

## 被停權成員的權限 (Permissions For Timed Out Members)

被停權成員（禁言成員）將暫時失去除了 `VIEW_CHANNEL` 和 `READ_MESSAGE_HISTORY` 之外的所有權限。擁有 `ADMINISTRATOR` 權限的伺服器擁有者與管理員除外。
