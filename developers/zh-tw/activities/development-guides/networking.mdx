## 互動活動代理伺服器考量 (Activity Proxy Considerations)

基於安全原因，所有網路流量皆會透過 Discord 代理伺服器 (Discord Proxy) 進行路由。其底層使用 Cloudflare Workers，並帶來以下限制：

#### WebTransport
雖然目前僅支援 WebSocket，但我們正與上游供應商合作以支援 WebTransport。

#### WebRTC
目前不支援 WebRTC。

## 建構完整網址 (Construct A Full URL)

在某些情況下，您可能需要使用完整網址而非相對路徑。網址的組成如下：
1. 您希望使用的協定 (Protocol)
2. 您的應用程式用戶端 ID (Client ID)
3. Discord 代理網域 (`discordsays.com`)
4. 您的資源路徑

#### 範例

```javascript
const protocol = `https`;
const clientId = '<您的 CLIENT ID>';
const proxyDomain = 'discordsays.com';
const resourcePath = '/foo/bar.jpg';
const url = new URL(`${protocol}://${clientId}.${proxyDomain}${resourcePath}`);
```

例如，假設應用程式用戶端 ID 為 `12345678`：
- 相對路徑：`/foo/bar.jpg`
- 完整網址：`https://12345678.discordsays.com/foo/bar.jpg`

## 使用外部資源 (Using External Resources)

Discord 中的互動活動是透過 Discord 代理伺服器進行「沙盒化」的。這樣做是為了隱藏使用者的 IP 位址，並阻止來自已知惡意端點的網址。為此，Discord 開發者入口網站中設有為您的應用程式 [設定網址映射 (URL Mappings)](/developers/zh-tw/activities/development-guides/local-development#url-mapping) 的區塊。

網址映射的一個邊緣情況是，第三方 NPM 模組或其他資源可能會參考外部（非沙盒化）的網址。

例如，如果您的應用程式中有一個 npm 模組嘗試向 https://foo.library.com 發送 HTTP 請求，該請求將會因 `blocked:csp` 錯誤而失敗。

要解決此限制，有幾種方法可以考慮：

- 分叉 (Fork) 該函式庫（以使用映射後的網址）
- 使用安裝後公用程式，例如 [patch-package](https://www.npmjs.com/package/patch-package)
- 使用我們的嵌入式 App SDK 中的 `patchUrlMappings` API

在上述情況下，我們建議使用 `patchUrlMappings` API，因為這可以讓您從非沙盒化的開發環境無縫過渡到生產環境。

此方法呼叫接受一個「映射 (mappings)」陣列，它會將任何外部網路請求轉換為您所定義的映射。

請參閱下方範例：

- 在此範例中，假設您有一個第三方函式庫會向 foo.com 發送 HTTP 請求。
- 在開發者入口網站中，建立如下映射：`/foo` -> `foo.com`
- 接著在程式碼中初始化 SDK 時，呼叫此函數。

```javascript
import {patchUrlMappings} from '@discord/embedded-app-sdk';
const isProd = process.env.NODE_ENV === 'production'; // 實際的開發/生產環境檢查方式可能因您而異

async function setupApp() {
  if (isProd) {
    patchUrlMappings([{prefix: '/foo', target: 'foo.com'}]);
  }
  // 在此之後開始初始化 App...
}
```

<Info>
注意：`patchUrlMappings` 會修改瀏覽器的 `fetch`、`WebSocket` 與 `XMLHttpRequest.prototype.open` 等全域變數。根據您使用的函式庫，使用此輔助函數可能會產生副作用。請僅在必要時使用。
</Info>

## 安全考量 (Security Considerations)

#### 信任用戶端資料

請勿將來自 Discord 用戶端的資料視為絕對真實。在應用程式本地端使用這些資料是可以的，但請假設任何來自 Discord 用戶端的資料都有可能被偽造。這包括關於目前使用者、其 Nitro 狀態、目前所在的頻道等資料。如果您需要以可信賴的方式獲取這些資訊，請在完成 OAuth2 流程後，使用您取得的使用者憑證 (User Token)，從您的應用程式伺服器端直接呼叫 Discord API。

此外，來自 Discord 用戶端的資料事前並未經過消毒 (Sanitize) 清理。例如使用者名稱和頻道名稱都是任意的使用者輸入。請務必清理這些字串或使用如 `.textContent`（例如）來安全地在 UI 中顯示它們。

#### 使用 Cookie

若要為您的互動活動設定 Cookie，以便在透過代理伺服器的網路請求中使用，請確保 Cookie 的網域 (Domain) 與您 App 的完整 `{clientId}.discordsays.com` 網域相符。您還需要明確地在 Cookie 上設定 `SameSite=None; Partitioned`。`SameSite=None` 是必須的，因為瀏覽器會拒絕在 iframe 內存取更嚴格限制級別的 Cookie。`Partitioned` 則會將該 Cookie 的使用限制在僅限 Discord 的 iframe 內。

請放心：由於內容安全政策 (CSP) 會限制請求僅發送到您自己 App 的代理伺服器，其他互動活動將無法使用您互動活動的 Cookie 發送請求。
