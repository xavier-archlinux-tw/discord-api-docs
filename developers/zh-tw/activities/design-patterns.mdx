在建置您的互動活動之前，以下是一些引導原則供您考量，以提供成功的使用者體驗。

---

## 指導原則

### 著重互動而非孤立
- 互動活動並非用於螢幕分享，因為我們已經內建了該功能！
- 給予使用者相互互動的方法。
- 使用者的動作應該會影響其他參與者的體驗。

### 著重表達而非單調
- 我們的使用者希望能創造難忘的時刻。
- 建立能讓使用者發揮創意、表達自我並以他們想要的方式與他人分享的體驗。

### 著重易用性 (Accessibility) 而非排他性
- 包容性是關鍵。
- 降低互動活動的入門門檻，並獎勵深入參與體驗的使用者。

---

## 提供「單人加強版 (Solo-Plus)」體驗

成功的體驗應兼顧小群組階段（「Plus」），同時在單人遊玩（「Solo」）的情況下也能兼具趣味性與可玩性。

### 單人或單人遊戲
- 雖然大多數人在 Discord 上會與一小群好友一起玩，但請考量那些單人玩也很有吸引力、多人玩更有趣的體驗！
- 如果您不支援單人遊玩，預計會有許多使用者自己點進來查看您的互動活動。請考量如何提供預覽，並吸引他們帶好友一起回來。

### 小群組階段 (Solo-Plus)
- 與單人體驗相比，小群組階段（3-8 人）的使用者參與度與留存率更高。
- 讓玩家能與好友一起加入，可以吸引他們持續回來遊玩。

### 大群組階段
- 雖然您可以對使用者設定「最大參與人數」建議，但唯一的實際限制是能加入語音通話的人數。
- 請注意當通話中有 25 人或更多人時，您的互動活動會有何表現。

## 使用者狀態與隱私

與他人一起進行互動活動時，讓每位玩家都能看到其他人的動作與狀態。

### 讓每位玩家看到其他人的動作與狀態
- 使用者喜歡參與在感覺活躍的空間中。
- If 使用者自訂了他們的伺服器暱稱或頭像，請在遊戲中使用該伺服器暱稱或頭像。
- 顯示使用者何時在語音通話中發言，或他們是處於活躍還是閒置狀態。

![Bobble League 中的對話氣泡](/images/activities/bobble-bash.png)

### 尊重使用者隱私
- 對於匹配來自不同伺服器之使用者的互動活動階段，使用者名稱與頭像預設應進行匿名化處理。
- Discord 的個人識別資訊僅應在使用者確認後才載入至互動活動中。
- 僅在能看見這些識別資訊的人已經透過伺服器或私訊 (DM) 擁有存取權時，才使用這些識別資訊。

## 交付高品質的體驗

讓您的應用程式快速、易於加入，並最大化趣味性，以推出深受大眾喜愛的產品。

### 帶給使用者驚喜與愉悅
- 驚喜與愉悅是指關心使用者體驗您作品的微小細節。
- 在他們最意想不到的時候加入適當的情感，以傳遞魔法。

![Bobble League](/images/activities/bobble-league.png)

### 盡可能降低載入時間
- 這可以讓 Discord 上龐大的行動端使用者更輕鬆地隨時加入或退出。
- 請參閱下方的 [品質與測試建議](/developers/zh-tw/activities/design-patterns#quality-and-testing) 以瞭解最低品質支援與測試建議。
- 請參閱下方的 [技術考量](/developers/zh-tw/activities/design-patterns#technical-considerations) 以瞭解如何劃分載入工作以及使用各種開發工具來縮短載入時間的建議。
- 考量桌上型電腦與行動裝置上不同的螢幕大小與方向，並確保 UI 元素適當地縮放。

### 支援隨時加入與隨時退出
- 互動活動的加入是無摩擦且容易發現的，因此您可以預期使用者會在體驗中途加入。給予這些使用者一些事情做，即使只是讓他們旁觀，直到他們能無障礙加入。同樣地，使用者可能會在未通知的情況下離開或閒置 (AFK)。請優雅地處理這些情況。
- 為已加入通話但尚未開始遊玩的使用者提供「旁觀 (Spectate)」功能。這對於有理想遊玩人數限制的互動活動也很有幫助。

![在您的互動活動中支援隨時加入與隨時退出](/images/activities/eights.png)

### 讓您的應用程式盡可能可用

#### 跨平台支援
- 支援桌面與行動裝置將能把您的使用者群擴展到最大。
- Discord 的裝置使用率在行動與桌面平台之間基本持平。

#### Discord 擁有全球受眾
- 考慮支援多種語言與文化，使您的應用程式更適合所有 Discord 使用者。

#### 實作邀請功能
- 嵌入式 App SDK 允許從互動活動內部向其他好友發送邀請。
- 有意圖地將邀請按鈕加入到畫面或流程中，而不僅僅是在啟動畫面和開始時。有意圖地提示邀請的範例包括：
  - 玩家離開互動活動階段時
  - 未達到最低參與者人數時（例如任何需要 2 人以上才能開始、或者多人玩更有趣的活動，又或者像西洋棋這類需要對手的遊戲。）

#### 實作分享功能
- Discord 是一個使用者互相交流的社交平台。分享與邀請能讓您的應用程式在這些共享空間中生存與互動，使其對當前未遊玩的每個人都可見且可存取。
- 嵌入式 App SDK 允許在互動活動中發送圖片或 GIF。
  - 分享能記錄趣味與難忘時刻的相片或 GIF，或一些值得炫耀的東西。不要為了宣傳活動而強行讓事物變得可分享。
  - 單純分享高分可能不太吸引人，但分享遊戲中非常精妙的一步，或是一幅共同繪製、創造回憶的畫作，都是很好的話題開端，並可能吸引他人加入。

![來自 Chess in the Park 的分享時刻](/images/activities/chess-victory.png)

#### 文字頻道中的互動活動
- 互動活動的使用者介面、文案與使用者流程不應依賴語音中的人來解釋、組織、澄清或說明活動如何運作。
- 所有控制項、CTA 和說明都應足夠清晰，以便在文字頻道中遊玩的夥伴（特別是首次使用者）能夠快速開始使用該應用程式，而無需透過語音交談來學習它。
- 依靠動態的首次使用者體驗和「如何遊玩」說明、彈出訊息、CTA 按鈕等，但注意不要讓文案過於雜亂。

- 避免在參與前設置令人望而卻步的門檻（如強制登入/付費牆），尤其是在使用者體驗的早期階段。
- 避免使用會對其他非付費玩家產生不公平優勢的付費解鎖（即「付費獲勝, Pay-to-Win」）。
- 利用 Discord 作為社交平台的優勢：尋找機會為使用者提供自訂外觀、美化主題、新頭像等。

## 遊戲設計考量

### 建置易學難精的遊戲
- 如果玩家以前從未玩過，當他們當下加入多人遊戲（語音/視訊）時，往往會感到壓力。
- 具有簡單快速參考控制方案且易於上手的遊戲，能幫助玩家融入體驗，而不會將他們丟進深水區。
- 易於理解的遊戲循環結合易於消化的控制通常效果最佳。如果必須透過教學才能理解遊戲，那麼在當下加入遊戲會令人望而生畏。

### 參與語音的 Discord 使用者是我們最活躍的使用者
- 當使用者與好友一起在語音通話中時，互動活動的參與度與留存率會增加。

### 考量現有的遊戲預期與套路
- 如果您正在開發一款遊戲，您更有可能吸引身為玩家的使用者。
- 利用您正在開發的類型遊戲的現有套路與預期。
- 迎合您的遊戲（或遊戲類型）所提供的玩家興趣，並支持這些興趣作為吸引他們回歸的手段。

### 當互動活動的最後一個參與者離開時，活動將會關閉
- 下次啟動互動活動時，即使是在相同的私訊或語音頻道中，它也會是一個不同的實例，並且可能會有不同的參與者。

<ManualAnchor id="if-a-game-is-launched-what-percentage-of-the-time-do-users-reach-different-phases-of-the-game" />
### 如果遊戲啟動了，使用者有多少比例的時間能到達不同的遊戲階段？
- 這包括「啟動 ➔ 開始」，因為這兩者之間可能會有很大的流失（特別是如果您不支援單人遊玩或小組人數限制）。
- 這也包括遊戲中的各個檢查點，包含開始遊戲的階段中有多少比例能到達遊戲終點！
- 如果您的互動活動可以以不同的設定開始遊戲，請查看哪些設定更受歡迎或更成功——您可能需要變更您的預設值！
  - 對於非遊戲類，您可能需要分析某個動作（例如在共同觀看/共同聆聽活動中排隊播放影片/歌曲）。
- 如果在某個點的流失率非常高，請看看能否找出原因或變更流程。

<ManualAnchor id="how-many-games-are-played-in-a-session" />
### 在同一次活動階段中玩了多少局遊戲？
- 每次活動階段玩更多局遊戲並不一定更好（您可能有一個旨在成為一局長遊戲的互動活動），但這是一個很好的基線來理解。
- 如果您預期在每次活動階段會看到很多重複遊玩，但實際卻沒有，那麼這值得深入探究以瞭解更多。

<ManualAnchor id="how-does-the-group-size-impact-various-key-metrics" />
### 小組人數如何影響各項關鍵指標？
- 例如人數較多時是否更有可能或更不可能到達遊戲終點？重新遊玩？等等。
- 這可以幫助您發現您的互動活動是否在不同的組別大小中存在非預期的弱點——也許如果人數太多遊戲就會拖得很長，或者如果只有兩個人就不夠吸引人。
- 並非每個互動活動都需要針對龐大的組別大小進行建置，但如果您提供了與 X 人一起玩的選項，那麼確保該體驗對所有參與者來說都是愉快的就是很好的做法！

## 共同觀看 / 共同聆聽 (Co-Watching / Co-Listening)

- 共同觀看與共同聆聽應該不僅僅是螢幕分享。考量一些創意方式，讓每位使用者的體驗都能受到其他參與者的影響。
- 考慮提供主持人控制項，以防止播放過於混亂。如果提供了主持人控制，請確保每個人都能以某種方式參與，即使只是向主持人推薦內容。
- 如果某些內容並非對活動中的每個人都可用（例如地理區域限制內容），請確保在內容排入隊列與播放時讓大家知道。播放的內容應與語音通話中的其他人保持同步。
- 熟悉的設計模式可以幫助人們立即參與到您的互動活動中。大多數 Discord 使用者對媒體瀏覽（Spotify、YouTube、Netflix 等）極為熟悉，如果您遵循這些模式，他們將能立即理解您的內容。
- 行動裝置的音量可以透過裝置內建的控制項（裝置側邊的音量按鈕）進行控制，並且在桌面、行動裝置與瀏覽器上都能按預期運作。

## 技術考量 (Technical Considerations)

### 針對 iframe 進行開發
- 請記住，您需要與 Discord 用戶端本身爭奪資源（CPU、記憶體與 GPU），因為在執行您的互動活動時，該用戶端仍在完整運行。
- 優先考量首次互動時間 (Time-to-First-Interaction)。
- 盡可能減少您互動活動中的 Web Assembly (WASM)。
  - 較舊的 iOS 裝置特別容易受到 WebKit 在互動活動使用的前幾分鐘內對 WASM 進行優化編譯的影響。這會在不同程度上導致明顯的卡頓、裝置發熱，並可能降低 Discord 的音訊與視訊品質。
- 基於安全原因，所有網路流量皆會透過 Discord 代理進行路由。
- 在開發者入口網站中為您的互動活動建立多個版本（例如開發、預備、生產環境）。這允許您展示並測試最後一個穩定版本，同時擁有開發與預備環境。

### 在 Unity 中開發
雖然在 Unity 中針對 iframe 進行開發是可行的，但可能需要對您遊戲的各個層面進行廣泛的微調，以滿足效能與品質標準。其他網頁原生 (Web-First) 的遊戲引擎通常開箱即用就具有更好的效能。

## 品質與測試 (Quality and Testing)

### 驗證使用者介面與使用者體驗
- 確保所有按鈕與 UI 元素皆正常運作，且在標準互動活動使用或遊戲過程中沒有任何阻礙性問題。
- 測試 Discord 使用者名稱的處理，確保它們能根據所選格式（Discord 用戶名稱或伺服器身分）正確顯示。

### 效能測試
- 評估效能，以確保其能滿足跨手機、平板電腦與電腦，以及多種作業系統的絕大多數使用者的需求。卡頓或緩慢不應顯著影響互動活動或遊戲體驗。

### 音訊測試
- 檢查音效與背景音樂是否能按預期運作，且音量設定功能正常。在語音通話中遊玩時，能聽見聲音非常重要。

### 驗證互動活動控制項
- 確認所有遊戲控制項（包含點擊、拖曳、按鍵控制、滑動與懸浮）皆按預期運作且適合該平台。
- 測試應用程式內教學與說明的準確性，以及其與當前活動功能和使用者平台的相關性（桌面或行動裝置）。

### 行動裝置 UI 考量
- 留意安全區域，避免特定角落的瀏海或相機鏡頭遮擋或切斷 UI 元素與按鈕，導致無法點擊。
- 注意 Android 的彎曲螢幕邊緣可能會使按鈕無法點選。
- 考量 iOS/Android 的滑動手勢是否會與位於螢幕底部的 UI 元素或按鈕產生衝突。
- 注意 Android 裝置的返回按鈕，確保其不與螢幕底部的元素或按鈕重疊，以防使用者無法點擊。
- 尊重平台定義的安全區域 (Safe Area)，以防止任何按鈕或內容被切斷或失效。

### 桌面版易用性指南 (Desktop Accessibility Guidelines)

- 允許使用者在桌面版上透過重複按 TAB 鍵在 Discord 按鈕與欄位之間循環，確保互動活動不會搶走焦點控制權。
- 實作 ESC 鍵的功能以關閉任何開啟中的活動內彈出視窗，提供一致的使用者體驗。
- 選擇具有高對比度且對色盲友善的顏色，特別是對於需要根據顏色進行區分的關鍵遊戲元素。
- 避免對遊戲中至關重要但僅有顏色差異的物件使用相似的顏色，這可能會對色盲使用者造成挑戰。
