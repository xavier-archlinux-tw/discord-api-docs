應用程式指令是與 Discord 用戶端中應用程式進行原生互動的方式。有 3 種指令類型可在不同的介面中存取：聊天輸入框、訊息的快顯功能表（點擊訊息右上角的選單或按右鍵），以及使用者的快顯功能表（右鍵點擊使用者）。

![顯示不同類型應用程式指令的用戶端介面](/images/command-types.png)

## 應用程式指令物件 (Application Command Object)

<ManualAnchor id="application-command-object-application-command-naming" />
###### 應用程式指令命名規約 (Application Command Naming)

`CHAT_INPUT` 指令名稱和指令選項名稱必須符合以下正規表示式：`^[-_'\p{L}\p{N}\p{sc=Deva}\p{sc=Thai}]{1,32}$`（需設置 unicode 旗標）。如果所使用的任何字母有小寫變體，您必須使用小寫。不具有小寫變體的字元和/或無大小寫區分的字母仍然是允許的。`USER` 和 `MESSAGE` 指令可以混合大小寫且可以包含空格。

<ManualAnchor id="application-command-object-application-command-structure" />
###### 應用程式指令結構 (Application Command Structure)

| 欄位 | 類型 | 說明 | 適用類型 |
|----------------------------|--------------------------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|---------------------|
| id                         | snowflake | 指令的唯一 ID | 所有類型 |
| type?                      | [command types](/developers/zh-tw/interactions/application-commands#application-command-object-application-command-types) 之一 | [指令類型](/developers/zh-tw/interactions/application-commands#application-command-object-application-command-types)，預設為 `1` | 所有類型 |
| application_id             | snowflake | 父應用程式的 ID | 所有類型 |
| guild_id?                  | snowflake | 指令的伺服器 ID（如果不是全域指令） | 所有類型 |
| name                       | string | [指令名稱](/developers/zh-tw/interactions/application-commands#application-command-object-application-command-naming)，長度為 1-32 個字元 | 所有類型 |
| name_localizations?        | ?dictionary with keys in [available locales](/developers/zh-tw/reference#locales) | `name` 欄位的本地化對照字典。值需遵循與 `name` 相同的限制 | 所有類型 |
| description                | string | `CHAT_INPUT` 指令的說明，長度為 1-100 個字元。對於 `USER` 和 `MESSAGE` 指令，此值需為空字串 | 所有類型 |
| description_localizations? | ?dictionary with keys in [available locales](/developers/zh-tw/reference#locales) | `description` 欄位的本地化對照字典。值需遵循與 `description` 相同的限制 | 所有類型 |
| options? \*                | array of [command options](/developers/zh-tw/interactions/application-commands#application-command-object-application-command-option-structure) | 指令的參數，最多 25 個 | CHAT_INPUT          |
| default_member_permissions | ?string | 以位元集表示的預設成員[權限設定](/developers/zh-tw/topics/permissions) | 所有類型 |
| dm_permission?             | boolean | **已棄用（請改用 `contexts`）**；指出該指令是否可在與 App 的私訊 (DM) 中使用，僅適用於全域範圍的指令。預設情況下，指令是顯示的。 | 所有類型 |
| default_permission?        | ?boolean | 不建議使用，此欄位即將被棄用。指出當 App 被加入伺服器時是否預設啟用該指令，預設為 `true` | 所有類型 |
| nsfw?                      | boolean | 指出該指令是否為[年齡限制指令](/developers/zh-tw/interactions/application-commands#agerestricted-commands)，預設為 `false` | 所有類型 |
| integration_types?         | list of [integration types](/developers/zh-tw/resources/application#application-object-application-integration-types) | 可使用該指令的[安裝上下文](/developers/zh-tw/resources/application#installation-context)（僅適用於全域範圍指令）。預設為您 App [配置的上下文](/developers/zh-tw/resources/application#setting-supported-installation-contexts) | 所有類型 |
| contexts?                  | ?list of [interaction context types](/developers/zh-tw/interactions/receiving-and-responding#interaction-object-interaction-context-types) | 可使用該指令的[互動上下文](/developers/zh-tw/interactions/receiving-and-responding#interaction-object-interaction-context-types)（僅限於全域範圍指令）。 | 所有類型 |
| version                    | snowflake | 當發生實質性記錄變更時更新的自動遞增版本識別碼 | 所有類型 |
| handler?                   | [command handler types](/developers/zh-tw/interactions/application-commands#application-command-object-entry-point-command-handler-types) 之一 | 決定該互動是由 App 的互動處理常式處理，還是由 Discord 直接處理 | PRIMARY_ENTRY_POINT |

\* `options` 僅能為類型為 `CHAT_INPUT` 的應用程式指令設定。

\* `handler` 僅能為類型為 `PRIMARY_ENTRY_POINT` 且帶有 `EMBED_LINKS` 旗標的應用程式指令設定（例如具有互動活動的應用程式）。

<Danger>
`default_permission` 即將被棄用。您可以改為將 `default_member_permissions` 設定為 `"0"`，以預設對管理員之外的所有人停用該指令，和/或使用 `contexts` 來停用與您 App 私訊中全域範圍指令的顯示。
</Danger>

<ManualAnchor id="application-command-object-application-command-types" />
###### 應用程式指令類型 (Application Command Types)

| 名稱 (Name)         | 類型 (Type) | 說明 (Description)                                                                                                       |
|---------------------|------|-------------------------------------------------------------------------------------------------------------------|
| CHAT_INPUT          | 1    | 斜線指令 (Slash commands)；當使用者輸入 `/` 時顯示的文字指令 |
| USER                | 2    | UI 導向的指令；當您右鍵點擊或點按使用者時顯示 |
| MESSAGE             | 3    | UI 導向的指令；當您右鍵點擊或點按訊息時顯示 |
| PRIMARY_ENTRY_POINT | 4    | UI 導向的指令；代表啟動應用程式 [互動活動 (Activity)](/developers/zh-tw/activities/overview) 的主要方式 |

|----------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------|-----------------------------------------------|
| type                       | [application command option type](/developers/zh-tw/interactions/application-commands#application-command-object-application-command-option-type) 之一 | 選項類型 | 所有類型 |

| name_localizations?        | ?dictionary with keys in [available locales](/developers/zh-tw/reference#locales) | `name` 欄位的本地化對照字典。值需遵循與 `name` 相同的限制 | 所有類型 |
| description                | string | 1-100 個字元的說明 | 所有類型 |

| max_value?                 | integer for `INTEGER` options, double for `NUMBER` options | 允許的最大值 | `INTEGER` , `NUMBER`                          |
| min_length?                | integer | 允許的最小長度（最小為 `0`，最大為 `6000`） | `STRING`                                      |
| max_length?                | integer | 允許的最大長度（最小為 `1`，最大為 `6000`） | `STRING`                                      |

## 授權您的應用程式 (Authorizing Your Application)

應用程式指令不依賴於伺服器中的 Bot 使用者；它們使用 [互動 (interactions)](/developers/zh-tw/interactions/receiving-and-responding) 模型。要在伺服器中建立指令，您的應用程式必須獲得 `applications.commands` 權限範圍的授權，該 Scope 可以單獨使用，但也自動包含在 `bot` 權限範圍中。

當請求此 Scope 時，我們「簡化」了 OAuth2 流程，類似於新增 Bot。您不需要完成整個流程、兌換權杖，或執行任何此類操作。

如果您的應用程式的指令運作不需要伺服器中的 Bot 使用者，**您不需要在網址中新增 bot 權限範圍或權限位元欄位**。

## 註冊指令 (Registering a Command)

<Info>
指令只能透過 HTTP 端點進行註冊。
</Info>

指令的範圍可以是全域性的，也可以是針對特定伺服器的。全域指令適用於新增您應用程式的每個伺服器。如果該 App 擁有與使用者共享共同伺服器的 Bot，則個別 App 的全域指令也可用於私訊中。

伺服器指令僅適用於您在建立時指定的伺服器。伺服器指令在私訊中不可用。在每個範圍（全域與伺服器）內，每個應用程式、每個類型的指令名稱都是唯一的。這意味著：

- 您的 App **不能**擁有兩個同名的全域 `CHAT_INPUT` 指令
- 您的 App **不能**在**同一個伺服器上**擁有兩個同名的伺服器 `CHAT_INPUT` 指令
- 您的 App **不能**擁有兩個同名的全域 `USER` 指令
- 您的 App **可以**擁有一個同名的全域與伺服器 `CHAT_INPUT` 指令
- 您的 App **可以**擁有一個同名的全域 `CHAT_INPUT` 與 `USER` 指令
- 您的 App **不能**擁有 `PRIMARY_ENTRY_POINT` 伺服器指令
- 多個 App **可以**擁有同名的指令

此列表並非詳盡無遺。一般而言，請記住指令名稱在每個應用程式、每個類型以及每個範圍（全域與伺服器）內必須是唯一的。

一個應用程式可以擁有以下數量的指令：

- 100 個全域 `CHAT_INPUT` 指令
- 15 個全域 `USER` 指令
- 15 個全域 `MESSAGE` 指令
- 1 個全域 `PRIMARY_ENTRY_POINT` 指令

對於除了 `PRIMARY_ENTRY_POINT` 之外的所有指令類型，您在每個伺服器中也可以擁有相同數量的伺服器專屬指令。

<Danger>
每個伺服器、每天有 200 個應用程式指令建立次數的全域速率限制
</Danger>

### 建立全域指令 (Making a Global Command)

全域指令在您應用程式的*所有*伺服器上均可用。

全域指令具有內置的讀取修復 (read-repair) 功能。這意味著如果您更新了某個全域指令，而使用者在該指令對他們進行更新之前嘗試使用它，Discord 將會執行內部版本檢查並拒絕該指令，並觸發該指令的重新載入。

要建立**全域**指令，請發送如下的 HTTP POST 呼叫：

```py
import requests

url = "https://discord.com/api/v10/applications/<my_application_id>/commands"

# 這是一個 CHAT_INPUT（斜線指令）範例，類型為 1
json = {
    "name": "blep",
    "type": 1,
    "description": "Send a random adorable animal photo",
    "options": [
        {
            "name": "animal",
            "description": "The type of animal",
            "type": 3,
            "required": True,
            "choices": [
                {
                    "name": "Dog",
                    "value": "animal_dog"
                },
                {
                    "name": "Cat",
                    "value": "animal_cat"
                },
                {
                    "name": "Penguin",
                    "value": "animal_penguin"
                }
            ]
        },
        {
            "name": "only_smol",
            "description": "Whether to show only baby animals",
            "type": 5,
            "required": False
        }
    ]
}

# 授權方面，您可以使用您的 Bot token
headers = {
    "Authorization": "Bot <my_bot_token>"
}

# 或者使用具有 applications.commands.update 範圍之 App 的用戶端憑證 token
headers = {
    "Authorization": "Bearer <my_credentials_token>"
}

r = requests.post(url, headers=headers, json=json)
```

### 建立伺服器指令 (Making a Guild Command)

伺服器指令僅在建立時指定的伺服器中可用。伺服器指令會**立即**更新。我們建議您使用伺服器指令進行快速測試，並在全域指令準備好供大眾使用時再行發布。

要建立**伺服器**指令，請發送類似的 HTTP POST 呼叫，但將其範圍限制在特定的 `guild_id`：

```py
import requests

url = "https://discord.com/api/v10/applications/<my_application_id>/guilds/<guild_id>/commands"

# 這是一個 USER 指令範例，類型為 2
json = {
    "name": "High Five",
    "type": 2
}

headers = {
    "Authorization": "Bot <my_bot_token>"
}

r = requests.post(url, headers=headers, json=json)
```

## 更新與刪除指令 (Updating and Deleting a Command)

您可以透過向指令端點發送 `DELETE` 和 `PATCH` 呼叫來刪除和更新指令。這些端點分別是：

- 用於全域指令的 `applications/<my_application_id>/commands/<command_id>`，或
- 用於伺服器指令的 `applications/<my_application_id>/guilds/<guild_id>/commands/<command_id>`

由於指令在類型和範圍內具有唯一的名稱，因此我們將新指令的 `POST` 請求視為更新或新增 (upsert)。這意味著**以已在您應用程式中使用的名稱建立新指令將會更新現有的指令**。

有關應用程式指令端點及其參數的詳細文件位於[端點章節](/developers/zh-tw/interactions/application-commands#endpoints)。

## 上下文環境 (Contexts)

指令在[應用程式指令物件](/developers/zh-tw/interactions/application-commands#application-command-object)上具有兩組上下文設定，可供您配置何時何地可以使用它：
- `integration_types` 定義了指令所支援的**[安裝上下文 (installation contexts)](/developers/zh-tw/interactions/application-commands#installation-context)**
- `contexts` 定義了指令可被使用的**[互動上下文 (interaction contexts)](/developers/zh-tw/interactions/application-commands#interaction-contexts)**

這兩類指令上下文的詳細資訊位於以下章節。

<Info>
上下文與安裝在伺服器中 App 的任何 [指令權限 (command permissions)](/developers/zh-tw/interactions/application-commands#permissions) 是不同的，且不會對其造成影響。
</Info>

### 安裝上下文 (Installation Context)

[安裝上下文](/developers/zh-tw/resources/application#installation-context)是您 App 安裝的位置 —— 安裝在伺服器、使用者，或兩者皆是。如果您的 App 同時支援這兩種安裝上下文，在某些情況下，您可能會希望 App 的某些指令僅適用於其中一種上下文。例如，您的 App 可能有一個 `/profile` 指令，僅在將其安裝到使用者帳戶時才有意義。

只要任何包含的上下文在 [應用程式層級已被支援](/developers/zh-tw/resources/application#setting-supported-installation-contexts)，即可在建立或更新指令時，使用 [`integration_types` 欄位](/developers/zh-tw/interactions/application-commands#application-command-object-application-command-structure) 來設定指令所支援的安裝上下文。

指令在 `integration_types` 中的設定值可能會影響該指令在哪個[互動上下文](/developers/zh-tw/interactions/application-commands#interaction-contexts)中可見。

### 互動上下文 (Interaction Contexts)

指令的互動上下文決定了它在 Discord 用戶端的何處可以使用，且可以藉由在建立或更新指令時設定 [`contexts` 欄位](/developers/zh-tw/interactions/application-commands#application-command-object-application-command-structure) 來配置。

有三種 [互動上下文類型](/developers/zh-tw/interactions/receiving-and-responding#interaction-object-interaction-context-types) 對應於不同的介面：`GUILD` (`0`)、`BOT_DM` (`1`) 以及 `PRIVATE_CHANNEL` (`2`)。然而，`PRIVATE_CHANNEL` 互動上下文僅對安裝到使用者的指令有意義（當該指令的 `integration_types` 包含 `USER_INSTALL` 時）。

## 權限 (Permissions)

應用程式指令權限允許您的 App 在伺服器中為最多 100 個使用者、身分組和頻道啟用或停用指令。如果使用者擁有必要的權限，他們也可以在用戶端中更新指令權限。

<Warning>
指令權限僅能使用 [Bearer token](/developers/zh-tw/topics/oauth2#client-credentials-grant) 進行更新。使用 Bot token 進行驗證將會導致錯誤。
</Warning>

可以使用 [`GET /applications/{application.id}/guilds/{guild.id}/commands/{command.id}/permissions`](/developers/zh-tw/interactions/application-commands#get-application-command-permissions) 端點來獲取指令的當前權限。回應將包含一個名為 `permissions` 的陣列，其中包含關聯的 ID 與權限類型。

可以使用 [`PUT /applications/{application.id}/guilds/{guild.id}/commands/{command.id}/permissions`](/developers/zh-tw/interactions/application-commands#edit-application-command-permissions) 端點來更新指令權限。要呼叫此端點，App 必須使用經具有足夠權限的使用者授權之 [`applications.commands.permissions.update`](/developers/zh-tw/topics/oauth2#shared-resources-oauth2-scopes) 範圍的 Bearer 權杖。要使他們的權限被視為足夠，**進行驗證的使用者**（非您的 App 或 Bot 使用者）必須滿足以下所有條件：
- 在編輯該指令所在的伺服器中擁有 [管理伺服器 (Manage Guild) 與管理身分組 (Manage Roles) 權限](/developers/zh-tw/topics/permissions)
- 具有執行正在編輯之指令的能力
- 擁有管理將受影響之資源（身分組、使用者和/或頻道，取決於 [權限類型](/developers/zh-tw/interactions/application-commands#application-command-permissions-object-application-command-permission-type)）的權限

### 同步與取消同步權限 (Syncing and Unsyncing Permissions)

可以在用戶端中透過導覽至 `伺服器設定` > `整合`，然後點擊已安裝 App 右側的 `管理` 來存取指令權限介面。在介面最上方，使用者可以編輯特定使用者、身分組或頻道的權限。預設情況下，這些最上層的權限會套用到 App 的所有指令。然而，每項權限也都可以取消同步，並為個別指令進行自訂，以提供更細緻的控制。

當特定指令的權限與最上層權限不同（即去同步）時，介面會向使用者顯示「未同步」。

### 應用程式指令權限物件 (Application Command Permissions Object)

<ManualAnchor id="application-command-permissions-object-guild-application-command-permissions-structure" />
###### 伺服器應用程式指令權限結構 (Guild Application Command Permissions Structure)

在伺服器中獲取 App 指令的權限時傳回。

| 欄位 | 類型 | 說明 |
|----------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------|------------------------------------------------------|
| id             | snowflake | 指令 ID 或應用程式 ID |
| application_id | snowflake | 指令所屬之應用程式 ID |
| guild_id       | snowflake | 伺服器 ID |
| permissions    | array of [application command permissions](/developers/zh-tw/interactions/application-commands#application-command-permissions-object-application-command-permissions-structure) | 指令在伺服器中的權限設定，最多 100 個 |

當 `id` 欄位是應用程式 ID 而非指令 ID 時，權限將套用到所有不包含明確覆寫設定的指令。

<ManualAnchor id="application-command-permissions-object-application-command-permissions-structure" />
###### 應用程式指令權限結構 (Application Command Permissions Structure)

應用程式指令權限允許您在伺服器中為特定使用者、身分組或頻道啟用或停用指令。

| 欄位 | 類型 | 說明 |
|------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| id         | snowflake | 身分組、使用者或頻道 ID。也可以是 [權限常數](/developers/zh-tw/interactions/application-commands#application-command-permissions-object-application-command-permissions-constants) |
| type       | [application command permission type](/developers/zh-tw/interactions/application-commands#application-command-permissions-object-application-command-permission-type) | 身分組 (`1`)、使用者 (`2`) 或頻道 (`3`) |
| permission | boolean | 允許為 `true`，拒絕為 `false` |

###### 應用程式指令權限常數 (Application Command Permissions Constants)

以下常數可用於指令權限 payload 的 `id` 欄位。

| 權限 | 數值 | 類型 | 說明 |
|--------------|----------------|-----------|-------------------------|
| `@everyone`  | `guild_id`     | snowflake | 伺服器中的所有成員 |
| 所有頻道 | `guild_id - 1` | snowflake | 伺服器中的所有頻道 |

<ManualAnchor id="application-command-permissions-object-application-command-permission-type" />
###### 應用程式指令權限類型 (Application Command Permission Type)

| 名稱 (Name) | 數值 (Value) |
|---------|-------|
| ROLE    | 1     |
| USER    | 2     |
| CHANNEL | 3     |

為了提供對指令的微調存取權限，所有類型的伺服器和全域指令都支援應用程式指令權限。具有 [必要權限](/developers/zh-tw/interactions/application-commands#permissions) 的伺服器成員和 App 可以允許或拒絕特定使用者和身分組使用某個指令，或者為整個頻道切換指令的啟用狀態。

類似於貼文/執行緒 [從父頻道繼承使用者與身分組權限](/developers/zh-tw/topics/threads#permissions) 的方式，頻道的任何指令權限都將套用到其所包含的貼文/執行緒中。

<Info>
如果您沒有使用某個指令的權限，它將不會顯示在指令選取器中。具有 Administrator 權限的成員可以使用所有指令。
</Info>

<ManualAnchor id="application-command-permissions-object-using-default-permissions" />
###### 使用預設權限 (Using Default Permissions)

預設權限可以在建立指令時，透過使用 `default_member_permissions` 和 `context` 欄位新增至指令中。新增預設權限不需要任何 Bearer token，因為它是在指令建立期間配置的，而不是針對特定的身分組、使用者或頻道。

建立指令時可以使用 `default_member_permissions` 欄位來設定使用者必須擁有的權限才能使用它。`default_member_permissions` 的值是按位 OR 運算後的 [權限旗標](/developers/zh-tw/topics/permissions#permissions-bitwise-permission-flags) 組合，序列化為字串。將其設定為 `"0"` 將禁止伺服器中的任何人使用該指令，除非配置了特定的覆寫或使用者具有管理員權限。

在設定全域指令的 [互動上下文](/developers/zh-tw/interactions/application-commands#interaction-contexts) 時，您也可以在 `contexts` 中包含 `BOT_DM` (`1`)，以控制是否可以在與您 App 的私訊中執行該指令。伺服器指令不支援 `BOT_DM` 互動上下文。

<ManualAnchor id="application-command-permissions-object-example-of-editing-permissions" />
###### 編輯權限範例 (Example of editing permissions)

例如，以下指令預設在任何伺服器中除了管理員之外，任何人都無法使用：

```json
{
    "name": "permissions_test",
    "description": "A test of default permissions",
    "type": 1,
    "default_member_permissions": "0"
}
```

或者，這將僅為具有 `MANAGE_GUILD` 權限的使用者啟用該指令：

```py
permissions = str(1 << 5)

command = {
    "name": "permissions_test",
    "description": "A test of default permissions",
    "type": 1,
    "default_member_permissions": permissions
}
```

以下將會對特定頻道停用某個指令：

```py
A_SPECIFIC_CHANNEL = "<channel_id>"
url = "https://discord.com/api/v10/applications/<application_id>/guilds/<my_guild_id>/commands/<my_command_id>/permissions"

json = {
    "permissions": [
        {
            "id": A_SPECIFIC_CHANNEL,
            "type": 3,
            "permission": False
        }
    ]
}

headers = {
    "Authorization": "Bearer <my_bearer_token>"
}

r = requests.put(url, headers=headers, json=json)
```

## 斜線指令 (Slash Commands)

斜線指令 —— 即 `CHAT_INPUT` 類型 —— 是應用程式指令的一種。它們由名稱、說明和一個 `options` 區塊組成，您可以將其視為函式的參數。名稱和說明可協助使用者在眾多指令中找到您的指令，而 `options` 則在使用者填寫您的指令時驗證其輸入。

斜線指令還可以具有群組與子指令，以進一步組織指令。稍後會有更多相關內容。

<Warning>
斜線指令的每個指令、其選項（包括子指令與群組）以及選擇的名稱、說明和值屬性組合起來最多隻能有 8000 個字元。當存在 [本地化欄位](/developers/zh-tw/interactions/application-commands#localization) 時，每個欄位僅有最長的本地化值（包括預設值）會計入大小限制中。
</Warning>

## 子指令與子指令群組 (Subcommands and Subcommand Groups)

<Warning>
目前，子指令與子指令群組全部顯示在指令瀏覽器的最上層。未來這可能會改變，以將它們作為巢狀自動完成選項包含進去。
</Warning>

對於那些希望建立更有組織且更複雜之指令群組的開發者而言，子指令和群組是最佳選擇。

**子指令 (Subcommands)** 藉由**指定指令或群組內的操作**來組織您的指令。

**子指令群組 (Subcommand Groups)** 藉由**將指令內具有相似操作或資源的子指令進行分組**來組織您的**子指令**。

這些並不是強制性的規則。您可以自由地以任何您喜歡的方式使用子指令和群組；這只是我們對它們的理解方式。

<Danger>
使用子指令或子指令群組將會使您的基本指令無法使用。如果您同時擁有 `/permissions add | remove` 作為子指令或子指令群組，您將無法發送基本的 `/permissions` 作為有效指令。
</Danger>

我們支援在群組內進行一層深度的巢狀結構，這意味著您的最上層指令可以包含子指令群組，而這些群組可以包含子指令。**這是唯一支援的巢狀形式。** 以下是一些視覺範例：

```
有效 (VALID)

command
|
|__ subcommand
|
|__ subcommand

----

有效 (VALID)

command
|
|__ subcommand-group
    |
    |__ subcommand
|
|__ subcommand-group
    |
    |__ subcommand

----

有效 (VALID)

command
|
|__ subcommand-group
    |
    |__ subcommand
|
|__ subcommand
```

```
無效 (INVALID)

command
|
|__ subcommand-group
    |
    |__ subcommand-group
|
|__ subcommand-group
    |
    |__ subcommand-group

----

無效 (INVALID)

command
|
|__ subcommand
    |
    |__ subcommand-group
|
|__ subcommand
    |
    |__ subcommand-group
```

### 範例逐步說明 (Example Walkthrough)

讓我們看一個範例。假設您運行一個管理 Bot。您想建立一個可以執行以下操作的 `/permissions` 指令：

- 獲取使用者或身分組的伺服器權限
- 獲取使用者或身分組在特定頻道上的權限
- 變更使用者或身分組的伺服器權限
- 變更使用者或身分組在特定頻道上的權限

我們將從定義 `/permissions` 的最上層資訊開始：

```js
{
    "name": "permissions",
    "description": "Get or edit permissions for a user or a role",
    "options": []
}
```

![一個沒有參數的指令。上面寫著 /permissions](/images/command.png)

現在我們有一個名為 `permissions` 的指令。我們希望這個指令能夠影響使用者和身分組。與其建立兩個獨立的指令，我們可以使用子指令群組。我們在這裡使用子指令群組是因為我們正在對相似資源上的指令進行分組：`user` 或 `role`。

```js
{
    "name": "permissions",
    "description": "Get or edit permissions for a user or a role",
    "options": [
        {
            "name": "user",
            "description": "Get or edit permissions for a user",
            "type": 2 // 2 是 SUB_COMMAND_GROUP 類型
        },
        {
            "name": "role",
            "description": "Get or edit permissions for a role",
            "type": 2
        }
    ]
}
```

您會注意到這樣的指令**不會顯示**在指令瀏覽器中。這是因為群組實際上是指令的「資料夾」，而我們建立了兩個空資料夾。所以讓我們繼續。

既然我們已經有效地建立了 `user` 和 `role` 的「資料夾」，我們希望能夠進行 `get` 和 `edit` 權限。在子指令群組內，我們可以為 `get` 和 `edit` 建立子指令：

```js
{
    "name": "permissions",
    "description": "Get or edit permissions for a user or a role",
    "options": [
        {
            "name": "user",
            "description": "Get or edit permissions for a user",
            "type": 2, // 2 是 SUB_COMMAND_GROUP 類型
            "options": [
                {
                    "name": "get",
                    "description": "Get permissions for a user",
                    "type": 1 // 1 是 SUB_COMMAND 類型
                },
                {
                    "name": "edit",
                    "description": "Edit permissions for a user",
                    "type": 1
                }
            ]
        },
        {
            "name": "role",
            "description": "Get or edit permissions for a role",
            "type": 2,
            "options": [
                {
                    "name": "get",
                    "description": "Get permissions for a role",
                    "type": 1
                },
                {
                    "name": "edit",
                    "description": "Edit permissions for a role",
                    "type": 1
                }
            ]
        }
    ]
}
```

![具有分組子指令的指令。上面寫著 /permissions user get](/images/command-with-groups-subcommands.png)

現在，我們需要一些參數！如果我們選擇了 `user`，我們需要能夠挑選一個使用者；如果我們選擇了 `role`，我們需要能夠挑選一個身分組。我們還希望能夠在伺服器級別權限和頻道特定權限之間進行選擇。為此，我們可以使用選填參數：

## 使用者指令 (User Commands)

使用者指令是顯示在使用者快顯功能表（右鍵點擊或觸按）上的應用程式指令。它們是為您的 App 顯示針對使用者的快速操作的極佳方式。它們不接受任何參數，並將在互動回應中傳回您所點擊或觸按的使用者。

<Warning>
使用者必須在他們啟動使用者指令的頻道中擁有傳送文字訊息的權限。如果他們沒有此權限，他們將會收到來自互動的「權限不足 (Permission Denied)」錯誤。
</Warning>

<Danger>
建立使用者指令時不允許使用 `description` 欄位。然而，為了避免對資料模型造成破壞性變更，獲取指令時 `description` 將會是一個**空字串**（而非 `null`）。
</Danger>

###### 使用者指令範例 (Example User Command)


## 進入點指令 (Entry Point Commands)

一個進入點指令是使用者從 [應用程式啟動器 (App Launcher)](https://support.discord.com/hc/articles/21334461140375-Using-Apps-on-Discord#h_01HRQSA6C8TRHS722P1H3HW1TV) 開啟 App 的 [互動活動 (Activity)](/developers/zh-tw/activities/overview) 的主要方式。

要使進入點指令對使用者可見，App 必須啟用了 [互動活動 (Activities)](/developers/zh-tw/activities/overview)。
![應用程式啟動器中的進入點指令](/images/command-entry-point.png)

<ManualAnchor id="entry-point-commands-example-entry-point-command" />
###### 進入點指令範例 (Example Entry Point Command)

```json
{
    "name": "launch",
    "description": "Launch Racing with Friends",
    "type": 4,
    "handler": 2
}
```

### 進入點處理常式 (Entry Point Handlers)

當使用者啟動 App 的進入點指令時，[`handler`](/developers/zh-tw/interactions/application-commands#application-command-object-application-command-structure) 的值將決定如何處理該互動：

- 對於 `APP_HANDLER` (`1`)，App 負責 [回應互動](/developers/zh-tw/interactions/receiving-and-responding#responding-to-an-interaction)。它可以透過使用 `LAUNCH_ACTIVITY`（類型 `12`）[互動回呼類型](/developers/zh-tw/interactions/receiving-and-responding#interaction-response-object-interaction-callback-type) 啟動 App 關聯的互動活動來做出回應，或者採取其他操作（例如在頻道中發送後續訊息）。
- 對於 `DISCORD_LAUNCH_ACTIVITY` (`2`)，Discord 將會自動處理該互動，啟動關聯的互動活動，並向啟動它的頻道發送訊息。

### 預設進入點指令 (Default Entry Point Command)

當您啟用互動活動時，系統會自動為您的 App 建立一個名為 "Launch" 的進入點指令，並將 `DISCORD_LAUNCH_ACTIVITY` (`2`) 設定為其 [進入點處理常式](/developers/zh-tw/interactions/application-commands#entry-point-handlers)。您可以透過呼叫 [獲取全域應用程式指令](/developers/zh-tw/interactions/application-commands##get-global-application-commands) 端點並尋找 "Launch" 指令，來獲取自動建立之指令的詳細資訊（例如其 ID）。

有關更新或取代預設進入點指令的詳細資訊，請參閱 [設定進入點指令指南](/developers/zh-tw/activities/development-guides/user-actions#setting-up-an-entry-point-command)。

## 自動完成 (Autocomplete)

自動完成互動功能允許您的應用程式在使用者輸入時，動態地向使用者傳回選項建議。

自動完成互動**可以為選項值傳回部分資料**。只要該輸入通過用戶端驗證，您的應用程式就會收到任何現有使用者輸入的部分資料。例如，您可能會收到部分字串，但不會收到無效的數字。使用者目前正在輸入的選項將會隨附一個 `focused: true` 的布林欄位發送，而使用者已經填寫的選項也會發送，但沒有 `focused` 欄位。這是一個特殊情況，由於使用者尚未填寫某些否則為必填的選項，這些選項此時可能不會存在。

<Warning>
此驗證**僅在用戶端進行**。
</Warning>

```json
{
  "type": 4,
  "data": {
    "id": "816437322781949972",
    "name": "airhorn",
    "type": 1,
    "version": "847194950382780532",
    "options": [
      {
        "type": 3,
        "name": "variant",
        "value": "data a user is typ",
        "focused": true
      }
    ]
  }
}
```

## 本地化語系 (Localization)

應用程式指令可以進行本地化，這將會使其根據用戶端選擇的語言使用本地化名稱與說明。這完全是選填的。透過在建立或更新應用程式指令時提交適當的 `name_localizations` 和 `description_localizations` 欄位，即可為指令、子指令與選項的名稱和說明，以及選項的名稱進行本地化。

應用程式指令可以是部分本地化的 —— 不需要提供所有 [可用的語系 (locales)](/developers/zh-tw/reference#locales)，同一個指令內的不同欄位也無需支援相同的一組語系。如果欄位的本地化對照字典中不存在某個語系，該語系的使用者將會看到該欄位的預設值。無需使用預設值填滿所有語系對照。任何與預設值相同的本地化數值都將被忽略。

本地化選項名稱受到額外限制，即它們必須與該指令的所有其他預設選項名稱不同，並且與該指令在該語系下的所有其他選項名稱不同。

當利用指令本地化功能時，您的用戶端接收到的互動 payload 仍將使用預設的指令、子指令和選項名稱。要本地化您的互動回應，您可以使用互動 payload 中的 `locale` 鍵來確定用戶端選擇的語言。

配備了本地化對照的應用程式指令可能如下所示：

```json
{
  "name": "birthday",
  "type": 1,
  "description": "Wish a friend a happy birthday",
  "name_localizations": {
    "zh-CN": "生日",
    "el": "γενέθλια"
  },
  "description_localizations": {
    "zh-CN": "祝你朋友生日快乐"
  },
  "options": [
    {
      "name": "age",
      "type": 4,
      "description": "Your friend's age",
      "name_localizations": {
        "zh-CN": "岁数"
      },
      "description_localizations": {
        "zh-CN": "你朋友的岁数"
      }
    }
  ]
}
```

### 語言語系退回機制 (Locale Fallbacks)

對於應用程式指令，如果使用者的語言語系不存在於本地化設定中，則存在內置的退回 (fallback) 機制。如果退回語系也缺失，它將使用預設值。

<Warning>
您應該確保在適當的語系 Key 中包含您的預設值，否則它可能會意外地使用退回值。例如，如果您的預設值是 `en-US`，但您沒有在本地化中指定 `en-US` 值，選取了 `en-US` 的使用者在指定了 `en-GB` 的情況下將會看到 `en-GB` 的值。例如，如果您的指令預設名稱為 "color"，而您的本地化僅將 `en-GB` 值指定為 "colour"，則 `en-US` 語系的使用者將會看到 "colour"，因為 `en-US` 的 Key 缺失了。
</Warning>

| 語言語系 | 退回語系 |
|--------|----------|
| en-US  | en-GB    |
| en-GB  | en-US    |
| es-419 | es-ES    |

### 獲取本地化指令 (Retrieving localized commands)

雖然大部分傳回應用程式指令物件的端點都會傳回 `name_localizations` 和 `description_localizations` 欄位，但有些端點預設不會傳回。這包括傳回應用程式所有伺服器或全域指令的 `GET` 端點。相反地，這些端點將會提供額外的 `name_localized` 或 `description_localized` 欄位，其中僅包含與請求者之語系相關的本地化值。（仍可以藉由提供適當的查詢參數來獲取完整的字典）。

例如，如果以 `zh-CN` 語系發送批次 `GET` 請求，且包含上述指令，則傳回的物件將如下所示：

```json
{
  "name": "birthday",
  "type": 1,
  "description": "Wish a friend a happy birthday",
  "name_localized": "生日",
  "description_localized": "祝你朋友生日快乐",
  "options": [
    {
      "name": "age",
      "type": 4,
      "description": "Your friend's age",
      "name_localized": "岁数",
      "description_localized": "你朋友的岁数"
    }
  ]
}
```

如果在本地化字典中找不到請求者的語系，則該欄位對應的 `name_localization` 或 `description_localization` 也將不會存在。

語系是藉由檢視 `X-Discord-Locale` 標頭來確定的，如果不存在，則檢視 `Accept-Language` 標頭，最後則是使用者設定的語系。

## 年齡限制指令 (Age-Restricted Commands)

包含年齡限制內容的指令應在建立或更新時將 [`nsfw` 欄位](/developers/zh-tw/interactions/application-commands#application-command-object-application-command-structure) 設定為 `true`。將指令標記為年齡限制將限制誰可以看到和存取該指令，以及可以從哪些頻道存取。

<Warning>
啟用了[探索功能 (discovery enabled)](https://support-dev.discord.com/hc/en-us/articles/9489299950487)（這是在應用程式目錄中顯示的必要條件）的 App 不能包含任何年齡限制的指令或內容。
</Warning>

### 使用年齡限制指令 (Using Age-Restricted Commands)

要使用年齡限制指令，使用者必須年滿 18 歲，且可以從以下位置之一存取該指令：
- [年齡限制頻道 (age-restricted channel)](https://support.discord.com/hc/articles/115000084051-Age-Restricted-Channels-and-Content) 或
- 在其使用者設定中[啟用年齡限制指令](https://support.discord.com/hc/en-us/articles/10123937946007) *之後* 與 App 的私訊。

有關存取與使用年齡限制指令的詳細資訊，請參閱[說明中心](https://support.discord.com/hc/en-us/articles/10123937946007)。
